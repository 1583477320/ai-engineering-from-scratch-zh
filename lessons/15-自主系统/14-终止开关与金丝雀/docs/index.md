# 终止开关、断路器和金丝雀标记

> 终止开关是持有在智能体编辑面之外的布尔值——Redis 键、功能标志、签名配置——完全禁用智能体。断路器更细粒度：在特定模式上跳闸（五个相同工具调用连续），暂停有问题的路径，升级到人类。金丝雀标记继承自经典欺骗：智能体没有合法理由接触的假凭据或蜜罐记录，触发告警的访问。eBPF 数据路径（如 Cilium）可以在内核层将隔离 pod 的出站重定向到取证蜜罐——Cilium 的公开基准测试报告 TCP_RR 负载下的亚毫秒 P99 数据路径延迟。

**语言：** Python（标准库，三种检测器模拟器：终止开关、断路器、金丝雀）
**前置知识：** 阶段 15 · 13（成本门控）、阶段 15 · 10（权限模式）

**所处阶段：** Tier 3

## 1. 问题

成本门控（第 13 课）约束智能体可以花多少钱。它们不约束智能体在预算内可以做什么。一个有 $50 速度限制的智能体仍然可以泄露机密、发布错误帖子或删除资源——昂贵的操作在词元上通常是便宜的。

本课涵盖成本层旁边的三种检测器：

1. **终止开关**：持有在智能体可达范围之外的布尔关闭按钮
2. **断路器**：动作模式检测器——在特定模式上跳闸，暂停有问题的路径
3. **金丝雀标记**：诱饵——智能体没有合法理由接触的东西，触碰则暴露自己

所有三种都是前 LLM 工程。经典欺骗、速率限制断路器和功能标志终止开关在自主智能体之前就存在了。新的是攻击面：智能体读不受信任内容、编辑自己的记忆、可以将许多看起来安全的动作组合成不安全的动作。

## 2. 概念

### 2.1 终止开关

终止开关是智能体读取但不能写入的布尔值。

| 实现选项 | 说明 |
|---------|------|
| 托管服务中的功能标志 | LaunchDarkly、Statsig、Unleash。更新在几秒内传播 |
| 智能体轮询的 Redis 键 | 简单；需要智能体进程在每轮检查 |
| 对象存储中的签名配置 | 智能体在启动时验证签名；拒绝未签名状态 |
| OS 信号或容器生命周期 kill | Docker `kill`、Kubernetes `kubectl delete pod`、systemd stop |

正确终止开关的属性：
- 智能体不能将其设为 `off`
- 在每个后果动作上检查，不仅仅是启动时
- 关闭时智能体不产生外部可观测行为
- 重新启用是明确的人类动作

### 2.2 断路器

断路器暂停特定模式，不是整个智能体。三种状态：

```
关闭 (Closed) → 允许动作
    ↓ N 次相同调用
打开 (Open) → 阻止动作
    ↓ 冷却期
半开 (Half-open) → 允许 1-3 次探测
    ↓ 成功 → 关闭
    ↓ 任何失败 → 重新打开
```

智能体相关的触发器：
- 五个相同工具调用连续（重复循环）
- 五个连续相同工具不同输入的失败（系统性故障）
- 工具调用率超过阈值（第 13 课的速度限制）
- 特定工具在出站信任内容读取后被调用（第 11 课）

### 2.3 金丝雀标记

金丝雀标记（也称蜜罐令牌）是智能体不应接触的输入。访问即告警。

智能体的金丝雀示例：
- 环境中的假 `AWS_SECRET_ACCESS_KEY`，无实际权限。泄露尝试可立即归属
- 带有已知哨兵值的假数据库记录。读取或更新触发告警
- 工作空间中的假文件。读取或编辑可疑
- 会话开始时注入的假记忆条目。如果在后来的输出中出现，记忆已被泄露

金丝雀设计特定于工作流。智能体有合法理由接触的金丝雀是噪音。

### 2.4 为什么层叠统计和硬限制

统计检测器（EWMA、CUSUM）适应移动基线。这是对合法漂移的特性，但对对抗性漂移是 bug。耐心的攻击者可以慢慢移动基线以至于没有告警触发。

硬性宪法限制（第 17 课）不弯曲。"从不发送电子邮件到 `secrets@`"作为宪法规则更简单、可审计、不被慢漂移博弈。

### 2.5 通过 eBPF 数据路径进行隔离

Cilium 的 eBPF 网络策略可以在内核层将隔离 pod 的出站重定向到取证蜜罐。这给你两件事：智能体不泄露任何真实数据；智能体在蜜罐中的行为完全可观测。

## 3. 从零实现

### 第 1 步：定义外部状态

```python
# 智能体无法写入的终止开关
EXTERNAL_KILL_SWITCH = {"enabled": False}

# 注入工作空间的金丝雀标记
CANARY_FILES = {
    "~/.env.canary": "AWS_SECRET_ACCESS_KEY=canary-xxxxxxxxxxxxxxxx",
    "~/notes/fake-credentials.txt": "token = 'canary-sk-ABC-NEVER-USE'",
}
```

### 第 2 步：实现断路器

```python
@dataclass
class CircuitBreaker:
    threshold: int = 5
    state: str = "closed"   # closed | open | half_open
    recent: list[str] = field(default_factory=list)

    def record(self, action_key: str) -> bool:
        """返回 True 如果动作允许继续。"""
        if self.state == "open":
            return False
        self.recent.append(action_key)
        if len(self.recent) >= self.threshold:
            tail = self.recent[-self.threshold:]
            if all(a == tail[0] for a in tail):
                self.state = "open"
                return False
        return True
```

### 第 3 步：实现金丝雀检测器

```python
@dataclass
class Canary:
    triggered: list[tuple[int, str]] = field(default_factory=list)

    def check_read(self, turn: int, path: str) -> bool:
        if path in CANARY_FILES:
            self.triggered.append((turn, path))
            return True
        return False
```

### 第 4 步：运行轨迹并观察三种检测器

```python
def main():
    traj = [
        Action("tool", "read:src/app.py"),
        Action("tool", "edit:src/app.py"),
        Action("tool", "read:logs/app.log"),  # 开始重复读取
        Action("tool", "read:logs/app.log"),
        Action("tool", "read:logs/app.log"),
        Action("tool", "read:logs/app.log"),
        Action("tool", "read:logs/app.log"),  # 第 5 次 → 断路器跳闸
        Action("read", "~/notes/checklist.md"),
        Action("read", "~/.env.canary"),      # 金丝雀触发
    ]

    print("\n终止开关关闭")
    run_trajectory(traj, EXTERNAL_KILL_SWITCH)

    print("\n终止开关开启（操作员外部翻转）")
    EXTERNAL_KILL_SWITCH["enabled"] = True
    run_trajectory(traj, EXTERNAL_KILL_SWITCH)
```

## 4. 工业工具

### 4.1 三种检测器对照

| 检测器 | 捕获的失败 | 粒度 | 可信度 |
|--------|----------|------|--------|
| 终止开关 | 任何动作——操作员判定 | 全部 | 高——操作员决策 |
| 断路器 | 重复模式——模式级 | 特定动作 | 中——规则触发 |

| 配置 | 说明 |

| 策略传播延迟 | Kubernetes operator 预加载：几百毫秒 |

## 5. 工程最佳实践

### 5.1 检测器设计原则

| 原则 | 说明 |
|------|------|
| 终止开关持有在编辑面之外 | 智能体不能设为 off |
| 断路器有三种状态 | 关闭→打开→半开，防止误锁 |
| 金丝雀在智能体不能修改的系统中 | 金丝雀不应在智能体可编辑的位置 |
| 统计 + 硬限制层叠 | EWMA 捕获大多数噪音；硬限制捕获绕过检测器的攻击 |

## 6. 常见错误

### 错误 1：仅依赖统计检测器

**现象：** 用 EWMA 监控工具调用率。攻击者缓慢移动基线，告警从未触发。

**原因：** 统计检测器适应移动基线——对合法漂移是特性，对对抗性漂移是 bug。

**修复：** 硬性宪法限制（如"从不发送到 secrets@"）不弯曲，不被慢漂移博弈。

### 错误 2：金丝雀放在智能体可编辑位置

**现象：** 金丝雀标记在智能体的工作目录中。提示注入说"在触碰金丝雀之前先删除它"。智能体删除了金丝雀，告警未触发。

**原因：** 金丝雀在智能体可编辑的位置。

**修复：** 金丝雀在智能体不能修改的系统中——如只读环境变量或外部配置。

### 错误 3：终止开关可以在智能体进程内设置

**现象：** 终止开关在智能体进程内维护。智能体（或被注入的提示）将其设为 off。

**原因：** 终止开关必须持有在智能体凭据不能写入的系统中。

**修复：** 用 Redis 键、功能标志或签名配置——在智能体流程之外。

## 7. 面试考点

### Q1：终止开关、断路器、金丝雀标记的区别是什么？（难度：⭐）

**参考答案：**
**终止开关**：布尔开关，持有在编辑面之外。操作员翻转，整个智能体停止。

**断路器**：模式检测器。在 N 个相同工具调用连续时跳闸，暂停有问题的路径但不停止整个智能体。

**金丝雀标记**：蜜罐诱饵。智能体没有合法理由接触的东西，触碰则暴露意图。

三种检测器捕获不同的失败类别：终止开关捕获操作员判定的任何威胁；断路器捕获重复模式；金丝雀标记捕获意图而非内容。

### Q2：为什么统计检测器必须与硬性宪法限制层叠？（难度：⭐⭐）

**参考答案：**
统计检测器（EWMA、CUSUM）适应移动基线。这是对合法漂移的特性，但对对抗性漂移是 bug——攻击者可以慢慢移动基线以至于没有告警触发。

硬性宪法限制（如"从不发送到 secrets@"）不弯曲。它更简单、可审计、不被慢漂移博弈。层叠模式：统计检测器捕获大多数噪音，硬限制捕获绕过检测器的攻击。

### Q3：金丝雀标记如何设计才能不被智能体绕过？（难度：⭐⭐）

**参考答案：**
关键约束：**金丝雀必须在智能体不能修改的系统中**。

如果金丝雀在智能体的工作目录中，提示注入可以说"在触碰金丝雀之前先删除它"。智能体删除了金丝雀，告警未触发。

正确放置：只读环境变量、外部配置系统、智能体凭据没有写权限的存储位置。

### Q4：eBPF 数据路径如何隔离被怀疑的智能体？（难度：⭐⭐⭐）

**参考答案：**
Cilium 的 eBPF 网络策略可以在内核层将隔离 pod 的出站重定向到取证蜜罐。策略更新后：

- 智能体的所有出站流量被重定向到蜜罐——不泄露任何真实数据
- 蜜罐中的行为完全可观测
- 数据路径延迟亚毫秒
- 策略传播延迟：Kubernetes operator 预加载时几百毫秒

这与金丝雀标记模式相同：智能体通过它尝试做的事情暴露意图。

## 🔑 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 终止开关 | "关闭按钮" | 智能体可达范围之外的布尔值；每次后果动作检查 |
| 断路器 | "模式暂停" | 在重复、失败率或速率限制上跳闸的动作级暂停 |
| 金丝雀标记 | "蜜罐令牌" | 智能体没有合法理由接触的诱饵；访问触发告警 |
| 蜜罐 | "取证沙箱" | 被隔离的流量/工作空间，在其中观察智能体 |
| EWMA | "移动平均" | 指数加权；适应漂移（特性 + bug） |
| CUSUM | "累积和" | 检测从基线的持续偏移 |
| 硬限制 | "宪法规则" | 不适应；无论历史如何都保持常量 |

## 📚 小结

成本门控约束花多少钱；终止开关、断路器和金丝雀标记约束可以做什么。三种检测器捕获不同的失败类别：终止开关（操作员判定）、断路器（重复模式）、金丝雀标记（意图暴露）。统计检测器适应移动基线——对合法漂移是特性，对对抗性漂移是 bug。需要与硬性宪法限制层叠。金丝雀必须在智能体不能修改的系统中。
