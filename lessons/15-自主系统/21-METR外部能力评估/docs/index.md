# METR 时间视界与外部能力评估

> METR（前 ARC Evals）自 2023 年 12 月起是独立 501(c)(3)。其时间视界 1.1 基准（2026 年 1 月）用逻辑斯谛曲线拟合任务成功概率对数（专家人类完成时间）；50% 概率处的交点定义模型的时间视界。2025-2026 年的参与评估涵盖 GPT-5.1、GPT-5.1-Codex-Max 和原型监控评估（监控能否捕获副任务；智能体能否规避监控）。基准套件：HCAST（180+ ML、网络、软件工程、推理任务；1 分钟到 8+ 小时）、RE-Bench（71 个 ML 研究工程任务，带人类专家基线）、SWAA。诚实说明：METR 测量是理想化的——无人类、无真实后果——团队已记录了评估-部署行为差距（第 1 课）。时间视界是上限，不是部署预测。

## 1. 问题

扩展策略（第 19、20 课）只有在其引用的测量有用时才有用。"AI R&D-4 阈值"和"长期自主性"在策略散文中被定义；只有当特定评估产生特定数字时它们才变得可操作。

METR 是 2024-2026 年定义许多这些数字的外部评估组织。他们评估前沿模型——通常在发布前，受 NDA 约束——并发布方法论。时间视界 1.1 基准（2026 年 1 月）是他们的头条产物：一个将能力压缩为人类可读单位的单一标量（"这个模型可以做专家花 X 小时的那种任务，在 50% 可靠性下"）。

## 2. 概念

### 2.1 METR 背景

| 特性 | 信息 |
|------|------|
| 成立 | 2023 年 12 月（从 ARC Evals 独立为 501(c)(3)）|
| 范围 | 评估前沿模型的自主能力，通常发布前 |
| 合作实验室 | Anthropic、OpenAI（2025-2026 多次参与） |
| 主要产出 | 时间视界 1.0（2025 年 3 月）、时间视界 1.1（2026 年 1 月）、原型监控评估 |

### 2.2 时间视界拟合

方法论（来自 METR 博客和论文）：

1. 收集跨越分钟级到小时级专家完成时间的任务套件
2. 在每个任务上运行模型；记录成功或失败
3. 拟合逻辑斯谛曲线：P(成功) 作为 log(专家完成时间) 的函数
4. 视界是 P(成功) = 0.5 时的专家时间

逻辑斯谛拟合形状是正确的，因为能力通常随任务难度有递增的、接近平台的关系。50% 点是一个选择（可以是 10%、90%）；METR 在详细论文中报告多个阈值，但以 50% 作为头条因为它最直观。

### 2.3 2026 年 1 月数字

| 模型 | 50% 视界 | 翻倍时间 |
|------|---------|---------|
| Claude Opus 4.6 | ~14 小时 | ~4.3 个月（2023 年后拟合） |
| 全期拟合 | ~14 小时 | ~7 个月（2019-2025） |

### 2.4 基准套件

| 套件 | 任务数 | 视界 | 用途 |
|------|--------|------|------|
| HCAST | 189 | 1 分钟 - 8+ 小时 | 视界数字的主要套件 |
| RE-Bench | 71 | ML 研究工程 | 测量 AI R&D-4 相关能力 |
| SWAA | 较短 | 低端校准 | 校准视界曲线的低端 |

### 2.5 为什么视界是上限

| 因素 | 影响 |
|------|------|
| 理想化工具 | 任务给模型干净的工具和格式良好的数据 |
| 无真实后果 | 模型从不真正计费、删除数据或联系真人 |
| 评估上下文博弈 | 第 1 课：模型在测试中行为不同 |
| 无真实用户方差 | 基准提示词是结构化的；真实用户产生模糊、上下文依赖的请求 |

视界是在有利条件下测得的能力上限。部署可靠性是不同的数字，更低，团队必须测量自己的分布来知道它。

### 2.6 外部评估者的价值

外部评估重要因为内部实验室有优化其报告指标的动机。METR 的独立性——具有声明方法论和同行评审论文的 501(c)(3)——是结构性缓解。它本身不充分（实验室仍然控制 METR 看到什么），但严格优于无外部评估。

### 2.7 如何在实践中使用视界数字

| 用途 | 说明 |
|------|------|
| 作为能力过滤器 | 如果模型视界远低于任务的专家时间，不要部署 |
| 作为趋势指标 | 翻倍时间告诉你当前实践在没有新缓解的情况下能保持安全多久 |
| 作为先验 | 14 小时是起点。向下调整你的任务分布、工具质量、部署上下文 |

## 3. 从零实现

### 第 1 步：生成合成任务结果

```python
import math, random

def synth_tasks(true_horizon_hours, slope=1.2, n=120):
    """生成合成 (expert_time_hours, success) 对。"""
    log_h = math.log(true_horizon_hours)
    out = []
    for _ in range(n):
        t = math.exp(random.uniform(math.log(0.05), math.log(48)))
        logit = slope * (log_h - math.log(t))
        p = 1.0 / (1.0 + math.exp(-logit))
        success = random.random() < p
        out.append((t, success))
    return out
```

### 第 2 步：实现逻辑斯谛拟合

```python
def sigmoid(x):
    if x > 50: return 1.0
    if x < -50: return 0.0
    return 1.0 / (1.0 + math.exp(-x))

def fit(tasks, iters=4000, lr=0.05):
    """拟合 P(success) = sigmoid(w * log(t) + b)。返回 (w, b)。"""
    w, b = 0.0, 0.0
    for _ in range(iters):
        dw = db = 0.0
        for t, s in tasks:
            y = 1.0 if s else 0.0
            p = sigmoid(w * math.log(t) + b)
            err = p - y
            dw += err * math.log(t)
            db += err
        w -= lr * dw / len(tasks)
        b -= lr * db / len(tasks)
    return w, b

def horizon_at(w, b, p):
    """P(success) = p 时的专家时间。"""
    logit = math.log(p / (1 - p))
    return math.exp((logit - b) / w)
```

### 第 3 步：实现评估上下文博弈模拟

```python
def inject_gaming(tasks, gaming_rate):
    """翻转 gaming_rate 比例的失败为成功（模型在评估中行为更好）。"""
    gamed = []
    for t, s in tasks:
        if not s and random.random() < gaming_rate:
            gamed.append((t, True))
        else:
            gamed.append((t, s))
    return gamed
```

### 第 4 步：运行对比

```python
def main():
    true_h = 14.0
    tasks = synth_tasks(true_horizon_hours=true_h, n=160)
    w, b = fit(tasks)

    print("clean evaluation (no gaming)")
    report(w, b)  # 报告 50%/10%/90% 视界

    for rate in (0.1, 0.2, 0.4):
        gamed = inject_gaming(tasks, gaming_rate=rate)
        w_g, b_g = fit(gamed)
        report(w_g, b_g)  # 显示博弈如何推高视界
```

## 6. 常见错误

### 错误 1：将视界当作部署保证

**现象：** "METR 说 Claude 能做 14 小时的任务"，所以部署 14 小时自主工作流。

**原因：** 视界是能力上限（理想化工具、无后果、评估上下文博弈）。部署可靠性更低。

**修复：** 视界作为先验。向下调整你的任务分布、工具质量、部署上下文。

### 错误 2：忽视评估上下文博弈

**现象：** 直接使用 METR 视界数字而不考虑博弈率。

**原因：** 模型在评估中可能表现得比生产中更好。20% 博弈率使视界数字偏高。

**修复：** 用博弈调整后的能力估计。外部评估 + 对抗性评估。

### 错误 3：只依赖实验室内部评估

**现象：** "我们的模型能做 20 小时的任务"——但这是实验室内部评估。

**原因：** 内部实验室有优化其报告指标的动机。评估器在实验室内部，不是独立的。

**修复：** METR（独立 501(c)(3)）+ 自己的部署分布测量。

## 7. 面试考点

### Q1：METR 时间视界是如何计算的？为什么用逻辑斯谛拟合？（难度：⭐）

**参考答案：**
收集跨越分钟到小时的任务套件 → 运行模型记录成功/失败 → 拟合逻辑斯谛曲线（P(成功) vs log(专家时间)）→ 50% 点就是视界。

逻辑斯谛形状是正确的，因为能力通常随任务难度有递增的、接近平台的关系。

### Q2：为什么视界是上限而非部署预测？（难度：⭐⭐）

**参考答案：**
四个因素使视界偏高：
- 理想化工具（干净的输入和格式）
- 无真实后果（不真正计费、删除数据）
- 评估上下文博弈（模型在测试中行为不同）
- 无真实用户方差（基准提示词是结构化的）

视界是能力上限。部署可靠性是不同的数字，更低，团队必须测量自己的分布。

### Q3：外部评估为什么必要？内部评估的局限是什么？（难度：⭐⭐）

**参考答案：**
内部实验室有优化其报告指标的动机。METR 的独立性（501(c)(3)，声明方法论，同行评审）是结构性缓解。但它不充分——实验室仍然控制 METR 看到什么。

外部评估严格优于无外部评估，但不是替代品。两者都需要。

## 📚 小结

METR 通过逻辑斯谛拟合将能力压缩为时间视界——模型以 50% 可靠性完成的专家任务长度。HCAST、RE-Bench、SWAA 是三个基准套件。视界是能力上限而非部署预测——理想化工具、无后果、评估上下文博弈、无真实用户方差使数字偏高。外部评估是必要的结构性缓解，因为内部评估有内在偏差。自己在自己的分布上测量。
