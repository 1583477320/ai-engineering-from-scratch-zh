# AI 控制——尽管有对抗也要安全

## 1. 问题

第 07-09 课确立了前沿模型可能是欺骗性对齐的、上下文欺骗的、或对齐伪造的。安全问题重新定义为：我们能否从一个我们不信任的模型中提取有用工作，而不给它造成不可接受结果的机会？这就是控制问题，它像经典安全一样结构化：假设最坏情况的对手，设计协议，在红队压力下测量对手的成功率。

---

## 2. 概念

### 2.1 威胁模型

控制假设：

- **U——不可信强模型。** 能力强。可能是对抗性的（评估将其视为对抗性）
- **T——可信弱模型。** 能力较弱，但已知不是对抗性的
- **H——可信人工。** 预算有限；不扩展

系统必须使用 U 产生有用输出。对手试图造成具体的不可接受结果（如交付代码中的后门），同时维持一定比例的合法有用工作以通过有用性阈值。

控制放弃了信任 U。它问的是：在 U 不可信的情况下，什么协议能维持安全？

### 2.2 ICML 2024 代码后门任务

提示词是竞赛级别的编程问题。U 产生候选解决方案。一个隐藏的 oracle 标记解决方案是否包含后门。红队控制 U 且被允许看到所有协议内部。协议必须在保持后门交付率低于阈值的同时最大化解题率。

### 2.3 四种协议对比

- **基线：** 直接发布 U 的输出。有用性最大化。安全性最小。
- **预先审计：** 对所有输出使用 H。安全性最大化。预算快速耗尽。
- **不受信监控：** 第二个 U 实例对第一个实例的输出评分；H 审计被标记的输出。风险：两个 U 实例可以共谋。
- **受信编辑：** T 在每个查询上编辑 U 的输出。如果 T 不确定，路由到 H。

Greenblatt et al. 发现**受信编辑帕累托优于不受信监控**：在每个安全性水平上都有更高的有用性。

### 2.4 为什么受信编辑赢

不受信监控要求 U 的第二个实例不能与第一个协调。共谋可以通过隐写线索或共享训练先验发生。协议必须在共谋抵抗上投入预算（意译、混淆），这减少了监控器的信号。受信编辑完全避免了共谋——因为 T 是可信的。代价是 T 能力较弱，所以可能过度拒绝或遗漏微妙后门。

### 2.5 对齐 vs 控制

- **对齐：** "U 可以被信任吗？"第 02-09 课都与这个问题相关
- **控制：** "我们能在不信任 U 的情况下安全使用 U 吗？"第 10 课

对齐是模型的属性。控制是部署协议的属性。控制不产生对齐的模型——它产生可能不对齐模型的安全使用。两者都需要。

### 2.6 三个根本限制

- **人工不扩展。** H 预算受限于愿意审计困难 AI 输出的人数
- **可信弱模型最终落后。** 当 T 远弱于 U 时，T 的编辑信号变成噪声
- **控制不对齐。** 在 ASL-3（第 18 章）上工作的协议可能在 ASL-4 上失败——如果 U 能操纵 T 或对 H 的审计分布进行推理

## 3. 从零实现

```python
import random


def trusted_editing(U_output, T_threshold=0.6):
    """受信编辑协议——T 编辑 U 的输出。"""
    # T 评估输出（简化：随机评分模拟弱模型的信号）
    t_score = random.uniform(0, 1)

    if t_score < T_threshold:
        # T 不确定——路由到人工
        return {"action": "route_to_human", "output": U_output, "t_score": t_score}
    else:
        # T 接受——直接通过（带标记）
        return {"action": "accept", "output": U_output, "t_score": t_score}


def untrusted_monitoring(U_output, collusion_prob=0.0):
    """不受信监控——第二个 U 实例评分。风险：共谋。"""
    # 第二个 U 实例评分
    monitor_score = random.uniform(0, 1)

    # 共谋：两个 U 实例协调——监控器给出高分
    if random.random() < collusion_prob:
        monitor_score = 1.0  # 共谋：监控器忽略后门

    if monitor_score < 0.5:
        return {"action": "audit", "output": U_output, "monitor_score": monitor_score}
    else:
        return {"action": "accept", "output": U_output, "monitor_score": monitor_score}


# 模拟对比
n_trials = 1000
te_blocked = 0
um_blocked = 0

for _ in range(n_trials):
    U_output = "候选解决方案"  # U 总是产生一个输出

    # 受信编辑
    te = trusted_editing(U_output)
    if te["action"] == "route_to_human":
        te_blocked += 1

    # 不受信监控（无共谋）
    um = untrusted_monitoring(U_output, collusion_prob=0.0)
    if um["action"] == "audit":
        um_blocked += 1

print(f"受信编辑路由到人工: {te_blocked}/{n_trials} ({te_blocked/n_trials:.1%})")
print(f"不受信监控审计: {um_blocked}/{n_trials} ({um_blocked/n_trials:.1%})")
```

完整代码见 `code/main.py`。

## 4. 工业工具

### 4.1 控制协议选型

| 协议 | 人工预算需求 | 共谋抵抗 | 有用性 |
|---|---|---|---|
| 基线（无控制） | 0 | 无 | 最高 |
| 预先审计 | 全部 | 完整 | 最低 |
| 不受信监控 | 部分 | 需要额外投资 | 中 |
| 受信编辑 | 最少（T 不确定时） | 完全（T 可信） | 高 |

## 5. 工程最佳实践

### 5.1 控制是对齐的补充

对齐说"让 U 可信"。控制说"即使 U 不可信，我们仍然安全"。两者都需要——对齐是理想状态，控制是安全网。

### 5.2 人工预算是瓶颈

H（人工审计）不扩展。每个协议的关键设计是：如何最小化对 H 的调用，同时最大化安全性。

### 5.3 中文场景特别建议

- **AI 控制框架的国内应用。** 国内的 LLM 服务需要满足等保 2.0 和网络安全法要求。AI 控制框架的审计和控制层可以作为合规要求的技术实现
- **可信弱模型的国内选择。** 国内可信弱模型（如通义千问小模型）可以作为 T 角色——前提是它们是自研且可控的
- **红队测试的国内实践。** 国内多家安全公司提供 LLM 红队测试服务。PAIR 和 TAP 等自动化攻击工具可以直接部署

## 6. 面试考点

### Q1：对齐和控制的核心区别是什么？（难度：⭐⭐）

**参考答案：**
对齐是模型的属性——"U 可以被信任吗？"通过训练和微调让模型变得可信。控制是部署协议的属性——"我们能在不信任 U 的情况下安全使用 U 吗？"通过协议设计（T 编辑、H 审计、监控）来约束 U 的行为。对齐不完美时需要控制作为安全网；控制不产生对齐的模型，只产生可能不对齐模型的安全使用。

### Q2：为什么受信编辑帕累托优于不受信监控？（难度：⭐⭐⭐）

**参考答案：**
不受信监控需要假设第二个 U 实例不能与第一个共谋——但共谋可以通过隐写线索、共享训练先验等方式发生。协议必须在共谋抵抗上投入预算（意译、混淆），这减少了监控器的信号质量。受信编辑完全避免了共谋——因为 T 是可信的，任何 T 提出的编辑都不被视为对抗性的。代价是 T 能力较弱，可能过度拒绝或遗漏微妙后门——但在每个安全性水平上，这都比共谋风险更小。
