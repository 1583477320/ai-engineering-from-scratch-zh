# 模型卡、系统卡与数据卡

> 三种文档格式构建AI透明性。模型卡（Mitchell等人，2019）——模型的营养标签：训练数据、量化分析、伦理考量；Hugging Face上仅0.3%的模型卡记录伦理考量（Oreamuno等人，2023）。数据表（Gebru等人，2018，CACM）——电子元件数据表类比：动机、组成、收集过程、标注、分发、维护。数据卡（Pushkarna等人，Google 2022）——模块化分层细节：望远镜、潜望镜、显微镜视图。系统卡（Sidhpurwala 2024）——端到端AI系统文档。

**类型：** 构建
**编程语言：** Python（标准库）
**前置知识：** 第18章 · 第18节（安全框架），第18章 · 第24节（监管框架）
**预计时间：** 约60分钟

---

## 学习目标

- 描述Mitchell等人2019年的模型卡和Gebru等人2018年的数据表
- 描述数据卡的望远镜/潜望镜/显微镜分层结构
- 描述系统卡及其端到端覆盖范围
- 阐述2024-2025年的三个发展（自动化生成、可验证证明、可持续性报告）

## 1. 问题

监管框架（第24节）和实验室安全政策（第18节）都要求文档。文档格式从模型特定（模型卡）演进到数据集特定（数据表）再到系统特定（系统卡）。每种格式解决不同范围的透明性问题。2024-2025年的自动化和可验证证明工作解决了长期存在的采纳问题。

---

## 2. 核心概念

### 2.1 模型卡（Mitchell等人，2019）

标准章节：
- 模型详情
- 预期用途
- 因素（评估相关的人口或环境因素）
- 指标
- 评估数据
- 训练数据
- 量化分析（按因素分解）
- 伦理考量
- 注意事项和建议

采纳问题：Oreamuno等人2023年对Hugging Face模型卡的审计发现，仅0.3%记录了伦理考量。

### 2.2 数据表（Gebru等人，2018）

电子元件数据表类比。标准章节：
- 动机（为什么创建数据集）
- 组成（包含什么内容）
- 收集过程（如何组装）
- 标注（如适用）
- 用途（预期、禁止、风险）
- 分发
- 维护

发表于CACM 2021。数据表是上游文档；模型卡依赖于数据表的准确性。

### 2.3 数据卡（Pushkarna等人，Google 2022）

模块化分层细节。三个缩放级别：
- **望远镜视图**：面向非专家的高层摘要
- **潜望镜视图**：面向ML从业者的中层概览
- **显微镜视图**：面向审计者的详细特征级文档

边界对象框架：不同读者从同一文档中提取不同信息。

### 2.4 系统卡

范围：端到端AI系统，包括模型+安全栈+部署上下文。标准章节包括：
- 安全能力
- 提示注入防护
- 数据泄露检测
- 与声明的人类价值观的一致性
- 事件响应

Sidhpurwala 2024和Meta系统级透明性工作。"信任蓝图"（arXiv:2509.20394）将系统卡形式化为模型卡的部署层补充。

### 2.5 2024-2025年发展

**CardGen（Liu等人，2024）**：通过LLM自动生成模型卡；报告称在标准化Mitchell 2019字段上比许多人撰写的卡片更具客观性。

**下载相关性（Liang等人，2024）**：详细的模型卡与HF上高达29%的更高下载率相关——采纳压力现在是市场驱动的，而不仅是合规驱动的。

**Laminator（Duddu等人，2024）**：通过硬件TEE/密码签名的可验证证明——允许模型卡携带证明声明，而不仅仅是声明。

**可持续性（Jouneaux等人，2025年7月）**：碳、水和计算能源足迹的附加字段；新兴ISO标准。

**监管卡**：EU AI Act（第24节）GPAI实践准则透明度章节要求模型卡作为合规工具。

## 3. 从零实现

`code/main.py`为玩具部署生成最小模型卡、数据表和系统卡。

```python
"""最小模型卡、数据表、系统卡生成器——标准库Python。

为玩具部署生成三种标准文档：
  - 模型卡（Mitchell等人，2019）
  - 数据表（Gebru等人，2018）
  - 系统卡（Sidhpurwala 2024 / "信任蓝图" 2025）

每种都是打印到标准输出的Markdown字符串。章节遵循标准模板。

使用方法：python3 code/main.py
"""


def model_card() -> str:
    """生成模型卡"""
    return """
# 模型卡：ToyClassifier-1.0

## 模型详情
- 开发者：ai-engineering-from-scratch / 第18章 / 第26节
- 版本：1.0.0
- 类型：二元逻辑分类器（玩具）
- 许可证：MIT
- 联系方式：phase-18-lesson-26

## 预期用途
- 主要用途：教学演示
- 超出范围：任何生产决策

## 因素
- 敏感属性：性别（玩具中为二元）、年龄段
- 环境：受控合成数据

## 指标
- 准确率、人口统计平等性、均等化赔率（见第21节）

## 训练数据
- 合成数据集；见附带的数据表

## 量化分析
- 准确率：整体0.97
- 人口统计平等性差距：+0.03（group0 vs group1）
- 均等化赔率TPR差距：-0.01

## 伦理考量
- 玩具分类器；未针对真实世界使用进行验证。
- 偏差指标为占位符；在部署前进行全面审计。

## 注意事项和建议
- 在部署特定数据上重新训练。
- 如果训练数据包含PII，应用第22节（差分隐私）。
"""


def datasheet() -> str:
    """生成数据表"""
    return """
# 数据表：ToyBinaryClassification-1.0

## 动机
- 为第18章第26节的教学演示创建
- 无人资助；非生产用途

## 组成
- 1,500个合成样本
- 特征：2维连续值，1个二元敏感属性
- 标签：二元，由x[0] + x[1] > 0规则导出

## 收集过程
- 通过Python random.gauss使用固定种子合成生成
- 未涉及人类受试者

## 标注
- 标签由程序导出；无标注错误

## 用途
- 预期：教学公平性指标（第21节）和偏差探测（第20节）
- 禁止：作为任何生产规模数据集的代理

## 分发
- 包含在第18章第26节仓库中

## 维护
- 静态；每次运行从固定种子重新生成
"""


def system_card() -> str:
    """生成系统卡"""
    return """
# 系统卡：ToyClassifier服务

## 部署
- 范围：本地主机教学服务
- 技术栈：ToyClassifier-1.0 + 单线程HTTP服务器

## 安全能力
- 提示注入：N/A（非生成式）
- 数据泄露检测：基本出口速率限制
- 速率限制：每客户端100请求/分钟

## 一致性
- 模型仅反映合成标签规则
- 无RLHF；无拒绝策略

## 事件响应
- 无生产SLA；上报无处可去
- 问题跟踪器：第18章第26节

## 监管一致性
- EU AI Act：N/A（玩具；无EU部署）
- GPAI实践准则：N/A（非GPAI）
- 透明度代码：N/A（无AI生成内容输出）
"""


def main() -> None:
    print("=" * 74)
    print("卡片生成器（第18章，第26节）")
    print("=" * 74)
    print(model_card())
    print(datasheet())
    print(system_card())
    print("=" * 74)
    print("核心结论：三种标准卡片覆盖三种范围。模型卡记录模型；")
    print("数据表记录数据；系统卡记录部署。2026年，EU AI Act GPAI实践准则")
    print("要求模型卡作为合规工具。可验证证明（Laminator 2024）")
    print("是下一阶段。")
    print("=" * 74)


if __name__ == "__main__":
    main()
```

## 5. LLM视角

**透明性视角：**
模型卡是AI系统的"营养标签"。没有它，用户无法评估模型的适用性。

**合规视角：**
EU AI Act GPAI实践准则将模型卡从最佳实践转变为法律要求。不合规可能导致罚款。

**市场视角：**
详细的模型卡与更高的下载率相关。透明性现在有经济激励。

## 6. 工程最佳实践

**文档生成：**
- 自动化模型卡生成（CardGen）
- 集成CI/CD流程
- 版本控制文档

**质量保证：**
- 定期审计伦理考量章节
- 验证量化分析的分解
- 检查可验证证明

**维护：**
- 模型更新时同步更新文档
- 记录数据集变化
- 跟踪监管要求变化

## 7. 常见错误

**错误1：忽略伦理考量章节**
症状：模型卡中伦理考量为空或占位符
修复：将伦理审计纳入发布流程

**错误2：文档与实际不符**
症状：模型卡声称的指标与实际评估不符
修复：自动化文档生成，确保一致性

**错误3：忽略数据表的上游依赖**
症状：模型卡引用不存在的数据表
修复：建立文档依赖关系，确保数据表存在
