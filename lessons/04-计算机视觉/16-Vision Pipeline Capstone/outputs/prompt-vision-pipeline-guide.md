# Vision Pipeline 设计指南提示词

## 用途

将此提示词发送给 LLM，分析你的视觉管线架构，找出接口脆弱点、性能瓶颈和可改进之处。

## 提示词模板

```markdown
你是一个资深计算机视觉系统架构师。请审查以下视觉管线的设计，给出评估和改进建议。

## 管线信息

- **应用场景**: [填写：如零售货架审计 / 自动驾驶 / 医疗影像筛查]
- **检测器模型**: [如 YOLOv8n / Mask R-CNN ResNet50-FPN]
- **分类器模型**: [如 ConvNeXt-Tiny / ResNet-18]
- **输入分辨率**: [如 640x640 / 动态尺寸]
- **目标延迟 SLA**: [如 < 100ms / < 500ms]
- **部署平台**: [如 NVIDIA Jetson Orin / AWS EC2 g4dn / 边缘 ARM 设备]
- **预计 QPS**: [如 10 / 100 / 1000]
- **数据契约格式**: [描述你使用的类型定义]

## 管线流程

按顺序列出每个阶段。例如：
1. HTTP 接收图像字节
2. JPEG/PNG 解码 -> PIL Image RGB
3. Resize + Normalize -> Tensor CHW float [0,1]
4. 检测器前向传播 -> boxes, scores, labels
5. NMS 过滤低分框
6. 对每个 box 裁剪 ROI 并 resize 到 224x224
7. 分类器批量前向传播 -> class probabilities
8. 聚合检测结果 + 分类结果 -> JSON 响应

## 需要审查的问题

### 1. 接口脆弱性分析
- 各阶段之间的数据类型/格式是否一致？
- 是否有未处理的边界情况（空检测、越界框、损坏图像）？
- 如果替换检测器为不同架构，哪些接口会断裂？

### 2. 性能瓶颈预测
- 在给定硬件上，哪个阶段最可能是瓶颈？
- 预估每个阶段的耗时范围
- 给出 3 个最高回报的优化建议

### 3. 可扩展性
- 当前批量策略能否支撑预期 QPS？
- 如果需要支持 10 倍 QPS，架构需要哪些改动？
- 是否应该引入微批处理？为什么？

### 4. 可靠性
- 列出所有可能的失败路径
- 每个失败路径的恢复策略是什么？
- 是否需要灰度发布 / A/B 测试机制？

请用表格形式汇总审查结果。
```

## 注意事项

- 保持分析具体化——避免"性能可以更好"这类模糊评价
- 必须考虑实际部署硬件的差异
- 对于生产环境，P0 级问题必须在上线前解决
- 参考课程的工业工具选型框架
```
