## 1. 问题

四种编排模式在 2026 年的框架中反复出现。每种模式解决不同类型的复杂性——但大多数应用不需要复杂的多智能体编排。**从简单开始——只在单智能体加五个工作流模式不够用时才添加拓扑。**

## 2. 概念

### 2.1 四种编排模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **监督者-工作者** | 一个监督者调度多个工作者 | 标准任务分配 |
| **群体/点对点** | 平等协作 | 开放式对话/辩论 |
| **层级** | 上下级关系 | 复杂任务分解 |
| **辩论** | 多方对抗 | 提高事实性 |

### 2.2 选择决策

```
需要多智能体协作吗？
├── 否 → 直接使用工作流（第 12 节）
├── 是 → 选择模式：
│   ├── 有明确的任务分配 → 监督者-工作者
│   ├── 平等协作 → 群体/点对点
│   ├── 复杂任务需要分解 → 层级
│   └── 需要多方验证 → 辩论
```

### 2.3 模式对比

| 方面 | 监督者-工作者 | 群体 | 层级 | 辩论 |
|------|------------|------|------|------|
| 控制 | 集中 | 去中心化 | 分层 | 对抗 |
| 通信 | 星型 | 全连接 | 树型 | 链式 |
| 容错 | 中 | 高 | 低 | 高 |
| Token 成本 | 中 | 高 | 低 | 高 |

## 3. 从零实现

### Step 1：监督者-工作者模式

```python
class SupervisorAgent:
    """监督者——调度工作者。"""
    def __init__(self, workers):
        self.workers = workers

    def dispatch(self, task):
        """分配任务给最合适的工作者。"""
        for worker in self.workers:
            if worker.can_handle(task):
                return worker.execute(task)
        return "无可用工作者"


class WorkerAgent:
    def __init__(self, name, skills):
        self.name = name
        self.skills = skills

    def can_handle(self, task):
        return any(skill in task.lower() for skill in self.skills)

    def execute(self, task):
        return f"[{self.name}] 完成: {task[:30]}"
```

### Step 2：层级编排

```python
class HierarchicalAgent:
    """层级编排——上级分解任务，下级执行。"""
    def __init__(self, subordinates):
        self.subordinates = subordinates

    def plan(self, task):
        """将任务分解给下级。"""
        steps = task.split("→") if "→" in task else [task]
        results = []
        for i, step in enumerate(steps):
            if i < len(self.subordinates):
                result = self.subordinates[i].execute(step.strip())
                results.append(result)
        return results
```

## 5. 工程最佳实践

### 5.1 选择原则

- **从简单开始**：单智能体 + 工作流 → 只在不够用时才加多智能体
- **监控成本**：多智能体 = 更多 LLM 调用 = 更高成本
- **故障隔离**：每个智能体应能独立失败而不影响其他

### 5.2 踩坑经验

- **过度复杂化**：简单任务用多智能体——引入不必要的复杂性
- **模式选错**：辩论任务用了监督者-工作者——效率低

---

## 6. 常见错误

### 错误 1：所有任务都用多智能体

**现象：** 简单任务也被过度工程化——成本和复杂度都高。

**修复：** 从直接 API 调用开始——只在单智能体不够用时才引入多智能体。

### 错误 2：忽略智能体间通信开销

**现象：** N 个智能体的通信成本 N² 增长。

**修复：** 选择合适的拓扑——层级或稀疏而非全连接。

## 7. 面试考点

### Q1：四种编排模式分别适用于什么场景？（难度：⭐⭐）

**参考答案：**
(1) **监督者-工作者**：有明确任务分配的标准任务；(2) **群体/点对点**：平等协作的开放任务；(3) **层级**：复杂任务需要分解为子任务；(4) **辩论**：需要多方验证以提高事实性。大多数应用不需要多智能体——从单智能体开始。

## 📚 小结

四种编排模式覆盖不同复杂度场景。从简单开始——单智能体 + 工作流通常够用。多智能体增加灵活性但增加成本和复杂度。选择取决于任务是否需要协作或多方验证。
