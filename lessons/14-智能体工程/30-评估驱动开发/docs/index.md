## 1. 问题

大多数团队在开发智能体后"再评估"——但那时问题已经定型了。评估应该是**外循环**——从第一天开始，持续指导每一步决策：提示词怎么写、工具选什么、什么时候需要多智能体。

---

## 2. 概念

### 2.1 三个评估层

| 层级 | 速度 | 成本 | 用途 |
|------|------|------|------|
| **静态基准** | 秒 | 低 | 提示词迭代 |
| **自定义离线** | 分钟 | 中 | 功能测试 |
| **在线生产** | 实时 | 高 | 质量监控 |

### 2.2 评估流水线

```
静态基准（秒级）→ 通过？→ 自定义离线（分钟级）→ 通过？→ 部署→ 在线监控（实时）
       ↓失败           ↓失败                   ↓异常
   修改提示词       调试智能体              回滚/修复
```

### 2.3 评估设计原则

- **从简单开始**：前 3 个测试用例就能发现 90% 的问题
- **覆盖边缘情况**：空输入、异常格式、长文本
- **自动回归测试**：每次修改都运行

## 3. 从零实现

### Step 1：三层评估框架

```python
import time

class EvalPipeline:
    """三层评估流水线。"""
    def __init__(self):
        self.static_tests = []
        self.offline_tests = []

    def add_static_test(self, name, fn, expected):
        self.static_tests.append({"name": name, "fn": fn, "expected": expected})

    def add_offline_test(self, name, fn, metric_fn):
        self.offline_tests.append({"name": name, "fn": fn, "metric": metric_fn})

    def run_static(self, agent):
        """秒级评估。"""
        pass_count = 0
        for test in self.static_tests:
            result = agent(test["fn"])
            if result == test["expected"]:
                pass_count += 1
        return f"静态: {pass_count}/{len(self.static_tests)} 通过"

    def run_offline(self, agent):
        """分钟级评估。"""
        pass_count = 0
        for test in self.offline_tests:
            result = agent(test["fn"])
            if test["metric"](result):
                pass_count += 1
        return f"离线: {pass_count}/{len(self.offline_tests)} 通过"
```

## 5. 工程最佳实践

- **评估不是最后一步**：它是外循环
- **从秒级开始**：前 3 个测试用例就够
- **迭代加速**：先静态→再离线→再在线

---

## 6. 常见错误

### 错误 1：等开发完再评估

**现象：** 上线前发现重大问题——需要重做。

**修复：** 从第一天开始评估——静态基准驱动每步迭代。

## 7. 面试考点

### Q1：为什么评估是外循环？（难度：⭐⭐）

**参考答案：**
评估驱动每一��决策——从提示词设计到工具选择到是否需要多智能体。放在最后意味着问题发现得太晚。三个层级：静态基准（秒级）、自定义离线（分钟级）、在线生产（实时）。
