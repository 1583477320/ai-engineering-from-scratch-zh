# 生产运行时：队列、事件、定时

> 生产智能体在六种运行时形态上运行：请求-响应、流式、持久执行、基于队列的后台、事件驱动、定时调度。在选择框架之前选择形态。可观测性在任何形态中都是关键。

**所处阶段：** Tier 3

## 1. 问题

开发阶段用 LangGraph 运行智能体——但生产环境需要不同的运行时形态。请求-响应适合对话，流式适合长生成，队列适合后台任务。选错形态→性能问题、成本失控、可观测性缺失。

---

## 2. 概念

### 2.1 六种运行时形态

| 形态 | 描述 | 适用场景 | 代表框架 |
|------|------|---------|---------|
| **请求-响应** | 同步、实时 | 聊天机器人 | FastAPI + LLM |
| **流式** | Server-Sent Events | 长文本生成 | SSE + stream=True |
| **持久执行** | 从故障点恢复 | 省时任务 | LangGraph |
| **基于队列** | 后台批量处理 | 数据处理 | Celery + Redis |
| **事件驱动** | 异步触发 | 工作流 | Kafka + Workers |
| **定时调度** | 定期执行 | 维护任务 | Cron + Agent |

### 2.2 形态选择决策

```
任务需要实时响应吗？
├── 是 → 请求-响应或流式
├── 否 → 任务能短期完成吗？
│   ├── 是 → 持久执行
│   ├── 否 → 有触发事件吗？
│   │   ├── 是 → 事件驱动
│   │   └── 否 → 定时或队列
```

### 2.3 可观测性矩阵

| 形态 | 延迟要求 | 可观测性关键指标 |
|------|---------|----------------|
| 请求-响应 | <3s | P50/P95 延迟、错误率 |
| 流式 | <100ms/TTFT | 首 token 时间 |
| 持久执行 | 分钟-小时 | 恢复次数、检查点大小 |
| 队列 | 分钟-小时 | 队列深度、处理速度 |
| 事件驱动 | 秒-分钟 | 事件丢失率 |
| 定时 | 小时-天 | 执行成功率 |

## 3. 从零实现

### Step 1：运行时选择器

```python
def select_runtime(task_type, latency_requirement="realtime"):
    """根据任务类型选择运行时形态。"""
    if latency_requirement == "realtime":
        if task_type in ["chat", "qa"]:
            return "request-response"
        elif task_type in ["generation"]:
            return "streaming"
    if latency_requirement == "batch":
        if task_type in ["data_processing"]:
            return "queue-based"
        elif task_type in ["maintenance"]:
            return "scheduled"
    if task_type in ["workflow"]:
        return "event-driven"
    return "durable-execution"
```

## 5. 工程最佳实践

- **先选形态再选框架**：形态决定架构
- **可观测性从第一天开始**：每种形态需要不同的指标
- **成本监控**：智能体的 LLM 调用成本可能失控

## 6. 常见错误

### 错误 1：所有任务都用请求-响应

**现象：** 长任务超时、用户体验差。

**修复：** 长任务用流式或队列——短任务用请求-响应。
