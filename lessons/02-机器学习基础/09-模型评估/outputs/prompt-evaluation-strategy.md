# 模型评估策略检查清单

面对一个新数据集和一个新模型时，按此顺序执行评估。每一步都有对应的常见陷阱。

## Step 1: 划定数据——在一切之前

- [ ] 将数据分割为训练集、验证集、测试集（常用比例 60/20/0 或 80/20，使用验证集调参时不可省略验证集）
- [ ] 分类任务必须使用**分层抽样**（Stratified Split），保证每个集合中各类别比例与全集一致
- [ ] 时间序列数据必须按时间先后切分，**禁止随机打乱后切分**
- [ ] 在测试集上**只评估一次**——每看一次测试集，它作为"无偏估计"的可靠性就降低一次
- [ ] 小数据集（<10K 样本）考虑使用交叉验证取代固定划分

## Step 2: 选基线——不能赢基线的模型不应上线

### 分类任务

| 基线 | 何时使用 |
|---|---|
| 多数类分类器（永远预测最多的类） | 通用起点 |
| 带类别权重的逻辑回归 | 当特征维度 > 样本数时 |
| 随机预测（按类别比例抽样） | 仅作为最低参照 |

### 回归任务

| 基线 | 何时使用 |
|---|---|
| 永远预测训练集目标均值 | 通用起点 |
| 线性回归（含偏置项） | 比均值基线略强，验证特征是否有效 |

## Step 3: 选指标——与业务目标对齐

### 分类

| 业务场景 | 首选指标 | 辅助指标 |
|---|---|---|
| 类别平衡，只要整体准确 | Accuracy | F1（查看宏平均） |
| 误报代价高（垃圾邮件、法律风险） | Precision | Precision@k |
| 漏报代价高（疾病、欺诈、安全） | Recall | F1、AUC |
| 类别不平衡且需要综合评估 | F1（macro 或 weighted） | AUC-ROC |
| 需要完整排序质量 | AUC-ROC | NDCG、mAP |

### 回归

| 业务场景 | 首选指标 |
|---|---|
| 需要对大误差敏感（MSE 的平方放大效应） | RMSE |
| 需要业务上可解释（"平均偏了多少钱"） | MAE |
| 需要对比不同量级的数据集 | R² |

## Step 4: 交叉验证

- [ ] 标准设置：5 折或 10 折
- [ ] 分类任务：**分层 K 折**
- [ ] 不稳定信号（标准差 > 均值的 5%）：考虑重复交叉验证（Repeated CV，如 5×2）
- [ ] 报告格式：`均值 ± 标准差`（单独报告均值会隐藏不稳定性）
- [ ] 检查每折的训练分数与验证分数的差距——差距大即过拟合

## Step 5: 统计检验——差异是否可信

当你发现模型 A 比模型 B 高 X% 时：

| 场景 | 推荐检验 | 显著性阈值 |
|---|---|---|
| 两模型在交叉验证的同一折上比较 | 配对 t 检验（paired t-test） | p < 0.05，\|t\| > t_critical(df=k-1) |
| 两模型在单次测试集上比较 | McNemar 检验 | χ² > 3.84 (df=1, p<0.05) |
| | 自助法（Bootstrap）置信区间 | 95% 置信区间不重叠 |

## Step 6: 诊断——学习曲线与偏差-方差

- [ ] 训练分高、验证分低且间隙大 → **高方差/过拟合** → 加数据、正则化、简化模型
- [ ] 训练分和验证分都低且接近 → **高偏差/欠拟合** → 增加特征、加模型复杂度
- [ ] 训练分低、验证分也低但训练更低 → **系统有问题** → 检查数据泄漏、模型是否在学习

## Step 7: 最终报告——要在上线前回答

```
1. 评估指标是什么？为什么选它？
2. 基线是多少？模型比基线好多少？
3. 这个差异经得起统计检验吗？
4. 验证集与测试集的差距大吗？（泛化能力）
5. 在不同类别/子集上的表现是否一致？
6. 是否有过拟合/欠拟合的迹象？
```

## 常见陷阱速查

| 陷阱 | 后果 | 避免方式 |
|---|---|---|
| 先分割再做预处理 | 数据泄露 | Pipeline + fit on train only |
| 只看准确率 | 被不平衡数据误导 | 看 F1、AUC、混淆矩阵 |
| 测试集被反复查看 | 测试集"退化为"验证集 | 建立物理隔离，只评估一次 |
| 时间数据随机切分 | 未来数据泄漏到训练 | TimeSeriesSplit |
| 用 2-3 折交叉验证 | 评估方差太大 | 至少 5 折 |
| 不看标准差 | 高得分可能只是噪声 | 报告 mean ± std |
