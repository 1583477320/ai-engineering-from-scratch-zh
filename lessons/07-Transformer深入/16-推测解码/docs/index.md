# 推测解码

> 用一个小模型快速"猜"多个词元，再用大模型并行验证——推测解码将大模型的生成速度提升 2-3 倍，而输出质量与完全用大模型生成完全相同。

**类型：** 概念课 | **语言：** Python | **前置知识：** 阶段 07 · 05（完整 Transformer）| **时间：** ~45 分钟

---

## 🎯 学习目标

- [ ] 理解推测解码的核心思想——小模型"提案"，大模型"验证"，正确接受、错误重采
- [ ] 说明为什么推测解码在输出质量上与完全用大模型生成等价
- [ ] 解释自验证——在不暴露大模型身份的情况下实现推测解码

## 1. 问题

LLM 推理是串行的——生成每个 token 都需要一次前向传播。对于 1.5B 参数的模型，在 A100 上生成 512 个 token 约需 0.5-1 秒。如果有一个 1.5B 的模型来快速"猜"多个 token，再让 70B 的大模型只验证一次——生成速度可以提升 2-3 倍。

---

## 2. 概念

### 2.1 推测解码的三步

```
小模型（草稿模型）生成 K 个候选 token:
  t1 = small_model(prompt)
  t2 = small_model(prompt + t1)
  t3 = small_model(prompt + t1 + t2)
  t4 = small_model(prompt + t1 + t2 + t3)

大模型（验证模型）并行验证 K 个 token:
  logits = big_model(prompt + [t1, t2, t3, t4])
  
  如果 logits[t1] 匹配 t1 → 接受
  如果 logits[t2] 匹配 t2 → 接受
  如果 logits[t3] ≠ t3 → 拒绝，从 logits 重新采样
```

### 2.2 为什么输出质量等价

**数学证明：** 推测解码的输出分布与完全用大模型生成的分布**完全相同**。关键在于拒绝采样——当小模型猜错时，从大模型的分布重新采样，保证最终分布不变。

### 2.3 每步节省的计算量

```
完全用大模型生成: 每步 1 次前向 = K 次前向
推测解码: K 次小模型前向 + 1 次大模型前向
小模型通常是大模型的 1/10-1/50 参数量
→ 总计算量降低 2-5 倍
```

---

## 🔑 关键术语

| 术语 | 含义 |
|---|---|
| 草稿模型 | 小型 LLM（如 0.5B），快速生成候选 token |
| 验证模型 | 大型 LLM（如 70B），并行验证多个 token |
| 接受/拒绝 | 草稿 token 与大模型分布匹配→接受；否则→从大模型分布重采样 |
| 自验证 | 草稿模型和验证模型是同一个模型——不需要额外的草稿模型 |

---

## 📚 小结

推测解码用小模型快速"猜"多个 token，再用大模型并行验证——正确接受，错误重采。关键洞察：输出分布与完全用大模型生成完全相同（数学证明）。在 1B→70B 的模型配对上，生成速度提升 2-3 倍。自验证让推测解码在没有单独草稿模型时也能工作。
