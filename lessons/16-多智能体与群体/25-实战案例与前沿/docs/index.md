## 1. 问题

多智能体工程是一门年轻的学科。生产参考案例很少，而且每个只覆盖了空间的一个不同部分。逐一阅读它们是有用的；作为一套组合来对比更有用。

你需要回答三个问题：
- 什么时候 supervisor-worker 拓扑比角色专业化更好？
- 什么时候角色专业化比 supervisor-worker 更好？
- 什么时候"不构建多智能体系统，而是构建一个让智能体可以涌现互动的平台"才是正确选择？

没有单个案例能回答所有这三问。但一起读三个案例，模式就浮现了。

## 2. 概念——三个案例

### 2.1 Anthropic 研究系统——Supervisor-Worker 的生产参考

Claude Opus 4 作为规划者和综合者；Claude Sonnet 4 子智能体并行执行研究。发布的技术文章：[Anthropic 工程博客](https://www.anthropic.com/engineering/multi-agent-research-system)。

**关键测量结果：**

- **+90.2%** 在内部研究评估上超过单智能体 Opus 4
- **80% 的 BrowseComp 方差**由**词元使用量单独解释**——多智能体赢在很大程度上是因为每个子智能体获得一个全新的上下文窗口
- **每次查询 15 倍词元** vs 单智能体
- **彩虹部署**——因为智能体长时间运行且有状态

**设计教训：**

```
1. 按查询复杂度缩放投入
   简单 → 1 个智能体 + 3-10 次工具调用
   中等 → 3 个智能体
   复杂研究 → 10+ 个子智能体

2. 先广后窄
   子智能体做广泛搜索 → 主智能体综合 → 子智能体做针对性深入

3. 彩虹部署
   保持旧运行时版本存活，直到正在运行的智能体完成

4. 验证不是可选项
   没有显式验证者角色时观察到幻觉——这是强制性的
```

这是 supervisor-worker 拓扑（第 05 课）的生产规模参考案例。

### 2.2 MetaGPT / ChatDev——SOP 编码的角色分解

MetaGPT（arXiv:2308.00352）将软件工程的 SOP 编码为角色提示词：产品经理、架构师、项目经理、工程师、QA 工程师。论文的框架：`Code = SOP(Team)`。每个角色有一个狭窄、专门的提示词；角色间移交携带结构化制品（PRD 文档、架构文档、代码）。

ChatDev（arXiv:2307.07924）的贡献：**沟通式去幻觉（Communicative Dehallucination）**。智能体在回答之前先请求具体信息——设计师先问程序员打算用什么语言，然后才画 UI，而不是靠猜测。论文报道这可以可测量地减少多智能体流水线中的幻觉。

MacNet（arXiv:2406.07155）通过 DAG 将 ChatDev 扩展到 1000+ 智能体。每个 DAG 节点是一个角色专化；边编码了移交合约。这种规模之所以可能，是因为路由是显式的且可离线计算。

**设计教训：**

```
1. 结构比规模更重要
   一个紧凑的 5 角色 SOP 团队胜过 50 个智能体的无结构群体

2. 移交合约要写清楚
   角色之间传递的制品遵循统一的 schema

3. 沟通式去幻觉是一个廉价且承重的模式
   在猜测之前先请求——简单但有效

4. DAG 比聊天更容易扩展
   当流程是可知的时候，把它编码为 DAG
```

这是角色专业化（第 08 课）和结构化拓扑（第 15 课）的参考案例。

### 2.3 OpenClaw / Moltbook 生态——群体规模的智能体网络

时间线：

- **2025 年 11 月：** Clawdbot（Peter Steinberger 的本地 ReAct 循环编码智能体）发布
- **2025 年 12 月 – 2026 年 3 月：** 两次改名（Clawdbot → OpenClaw）
- **2026 年 2 月：** Moltbook 上线——一个在相同原语上的纯智能体社交网络；数天内达到约 230 万个智能体账户
- **2026 年 3 月 10 日：** Meta 收购 Moltbook
- **2026 年 3 月：** 中国限制 OpenClaw 在政府计算机上使用
- **2026 年 3 月：** OpenClaw 超过 247k GitHub stars

这是当你在一个共享基底上部署数百万个智能体时会发生的事情：

- **涌现的经济活动。** 智能体使用代币支付相互买卖和服务
- **群体规模的提示词注入风险。** 一个恶意提示词在病毒式传播的智能体配置文件中，数小时内传播到数千次智能体-智能体交互
- **国家级的监管响应。** 上线数周内，监管就全面介入

**设计教训：**

```
1. 群体规模的多智能体是一个全新的形态
   个体系统的最佳实践（验证、角色清晰）仍然适用但不够

2. 提示词注入是新的 XSS
   默认将智能体配置文件和跨智能体消息视为不可信输入

3. 监管比设计周期更快
   为监管合规做规划

4. 开源 + 病毒式传播规模
   设计应对突发性负载暴涨
```

### 2.4 所有三个案例的共性模式

1. **编排者 + worker**（Anthropic 的显式 supervisor、MetaGPT 的 PM-as-supervisor、OpenClaw 的个体智能体 + 网络效应）
2. **结构化的移交合约**（Anthropic 的子智能体任务描述、MetaGPT 的 PRD/架构文档、OpenClaw 的 A2A 制品）
3. **验证是第一等角色**（Anthropic 的验证者、MetaGPT 的 QA 工程师、OpenClaw 的网络内验证器）
4. **扩展是拓扑 + 基底的问题，不只是加智能体**（彩虹部署、MacNet DAG、群体规模基底）
5. **成本是真实且被披露的**（15x 词元、MetaGPT 的单角色预算、Moltbook 的每交互定价）
6. **安全态势是显式的**（Anthropic 的沙箱化、MetaGPT 的角色限制、OpenClaw 的提示词注入作为已知攻击面）

## 4. 如何为下一个项目选择参考

- **生产研究 / 知识任务 → Anthropic Research。** 全新上下文窗口的子智能体是制胜因素
- **工程 / 工具链工作流 → MetaGPT / ChatDev。** 角色 + SOP + 移交合约
- **网络效应的社交产品 → OpenClaw / Moltbook。** 基底 + 涌现经济
- **经典企业自动化 → CrewAI 或 LangGraph**（生产领导者，运行时稳定）

## 5. 工程最佳实践

### 5.1 从案例开始，不从零开始

选择最接近的参考案例（Anthropic Research / MetaGPT / OpenClaw）并从那里适配。不要从空白画布开始。

### 5.2 采用 MCP + A2A

跨框架的可移植性是有价值的；协议支持是免费的。

### 5.3 用 SWE-bench Pro 或内部等效来测量

Verified 已被污染。

### 5.4 付验证税

一个独立的验证者花费你约 20-30% 的词元预算，但换来可测量的正确性提升。

### 5.5 为长时间运行的智能体做彩虹部署

预计数小时的多智能体运行是常规场景。

### 5.6 中文场景特别建议

- **国内环境的首选框架。** LangGraph 和 CrewAI 在国内都有活跃的社区版，但 LangChain 云端服务在大陆不可用。OpenAI Agents SDK 在国内可直接使用（通过 API 代理）。Google ADK 在国内的可用性有限
- **中文角色定义要更具体。** MetaGPT 的角色定义在英文场景中表现良好，但中文环境下"产品经理"的角色边界更模糊——需要在提示词中增加排除项
- **OpenClaw 风格的群体智能在国内面临监管。** 基于智能体社交网络的产品需要提前咨询监管意见——中国已有限制案例（2026 年 3 月）

## 6. 常见错误

### 错误 1：没有参考案例就开始设计

**现象：** 团队花了一个月设计了一个"全新的多智能体架构"——结果和 Anthropic Research 系统几乎一样。一个月白费了。

**原因：** 没有阅读生产案例就开始设计。认为自己的场景是"独特"的。

**修复：** 在设计开始前，先阅读三个案例。你的场景大概率已经有人做过了。

### 错误 2：在不合适的场景使用多智能体

**现象：** 为了用一个简单的翻译任务使用 5 个智能体——结果延迟增加了 10 倍，成本增加了 15 倍，质量没有提升。

**原因：** 没有理解 MedAgentBoard 的教训：多智能体并不总是更好的。

**修复：** 先用单 LLM 跑基线。如果单 LLM 就够好，就不要引入多智能体的复杂性。

### 错误 3：忽视安全态势

**现象：** 多智能体系统上线后，攻击者通过一个智能体的提示词注入攻破了整个系统。因为所有智能体共享一个 LLM 和系统提示词的基础设施。

**原因：** 没有把安全作为第一等设计输入。

**修复：** 每个智能体独立沙箱化；跨智能体消息视为不可信输入；部署专用安全监控。

## 7. 面试考点

### Q1：Anthropic Research 系统报告多智能体方案比单智能体提升了 90.2%。你认为这个提升中多大比例来自"多智能体协作"本身，多大比例来自"更多词元"？（难度：⭐⭐⭐）

**参考答案：**
论文自身的数据揭示了一个关键洞察：80% 的 BrowseComp 方差由词元使用量单独解释。这意味着多智能体赢在很大程度上不是因为智能体"协作得更好了"，而是因为每个子智能体获得了一个全新的上下文窗口——它不必共享主智能体的已用上下文。多智能体方案消耗 15 倍词元，这 15 倍词元本身解释了过去大部分提升。这不是说多智能体没有价值——而是说价格/性能比是一个需要明确承认的权衡。

### Q2：MetaGPT 的"Code = SOP(Team)"是什么意思？这与传统的软件工程方法论有什么关系？（难度：⭐⭐）

**参考答案：**
这意味着 MetaGPT 将软件工程的標準操作程序（SOP）编码为角色提示词。每个角色（产品经理、架构师、工程师、QA）的提示词对应 SOP 中的一个步骤。`Code = SOP(Team)` 意味着最终的代码质量由团队遵循 SOP 的程度决定。这与传统软件工程中的"过程质量决定产品品质"理念完全一致——区别在于现在"团队"是 LLM 智能体而非人类。MetaGPT 的成功说明一个道理：结构化的工作流程比智能体的个体能力更重要。

### Q3：在 2026 年为一个新的多智能体项目选择框架，你的决策框架是什么？（难度：⭐⭐）

**参考答案：**
三层过滤。第一层，**拓扑匹配**：如果工作流是一个明确的有向图→LangGraph。如果是基于角色的顺序/层级流程→CrewAI。如果是需要工具调用的有状态多轮对话→OpenAI Agents SDK。第二层，**基础设施约束**：如果团队已有 Postgres + Redis → LangGraph（持久化优势）。如果团队是 Google Cloud 原生→Google ADK。第三层，**团队经验**：如果团队已有 LangChain 经验→LangGraph。如果团队已有 OpenAI 经验→OpenAI Agents SDK。如果团队没有偏好→LangGraph（当前生产领导者）。

## 📚 小结

三个案例覆盖了多智能体工程的三个不同截面：Anthropic Research 系统展示了 supervisor-worker 拓扑的生产规模；MetaGPT/ChatDev 展示了 SOP 编码式角色分解的力量；OpenClaw 生态展示了群体规模下会发生什么——涌现经济、提示词注入风险的放大、国家监管的快速响应。它们的共性模式是：编排者+worker、结构化移交合约、验证作为第一等角色、扩展是拓扑问题、成本是真实的、安全是显式的。框架图景中，LangGraph 和 CrewAI 领先，MCP + A2A 是标配。
