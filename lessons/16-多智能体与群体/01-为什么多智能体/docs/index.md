# 为什么需要多智能体？

> 一个智能体碰到墙时，正确的做法不是造一个更大的智能体——而是造更多智能体。

## 1. 问题

你在阶段 14 构建了一个单智能体。它工作了。你可以读文件、运行命令、调用 API、推理结果。然后你把它指向一个真实的代码库：200 个文件、三种语言、依赖基础设施的测试、以及在写代码之前需要研究外部 API 的要求。

智能体卡住了。不是因为 LLM 笨，而是因为任务超出了一个智能体循环能处理的范围。上下文窗口被文件内容填满。智能体忘了 40 次工具调用前读到的内容。它试图同时做研究员、编码者和审查员，三个都做得很差。

## 2. 概念

### 2.1 单智能体天花板

```
┌─────────────────────────────────────────┐
│            单 智 能 体                   │
│  ┌───────────────────────────────────┐  │
│  │         上下文窗口                 │  │
│  │  研究笔记 + 代码 + 测试输出       │  │
│  │  + 审查反馈 + API 文档 + ...      │  │
│  │  ██████████████████████ 满了 ███ │  │
│  └───────────────────────────────────┘  │
│  一个系统提示词试图覆盖                  │
│  研究 + 编码 + 审查 + 测试               │
│  结果：每样都做得平庸                    │
└─────────────────────────────────────────┘
```

三件事崩溃：
1. **上下文饱和**——到第 30 轮，已消耗 150K 词元。第 5 轮的关键细节丢失
2. **角色混乱**——"你是研究员、编码者、审查员和测试者"产生一个半做每件事的智能体
3. **顺序瓶颈**——读文件 A，然后 B，然后 C。三次串行 LLM 调用。无并行

### 2.2 多智能体方案

拆分工作。给每个智能体一个任务、一个上下文窗口、一个为该任务调优的系统提示词：

```
                     编排者
        ┌──────────┬──────────┬──────────┐
        ▼          ▼          ▼          ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
   │研究员    │ │编码者    │ │审查员    │ │测试者    │
   │读文档    │ │写代码    │ │检查代码 │ │运行测试 │
   │找模式    │ │基于研究  │ │找 bug   │ │报告结果 │
   └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘
        └──────────┴───────────┴───────────┘
                       合并结果
```

每个智能体拥有：专注的系统提示词、自己的上下文窗口、清晰的输入/输出契约。

### 2.3 真实系统中的多智能体

| 系统 | 架构 | 特性 |
|------|------|------|
| Claude Code 子智能体 | 父→子任务委派 | 父保持上下文干净，子做专注工作 |
| Devin | 规划者+编码者+浏览器 | 三个独立上下文 |
| SWE-bench 顶级系统 | 研究者+规划者+编码者 | 单智能体得分更低 |
| ChatGPT Deep Research | 多搜索智能体并行 | 不同角度搜索后合成 |

### 2.4 四种编排模式

**模式 1：流水线**
```
输入 → 研究者 → 编码者 → 审查员 → 输出
```
每个智能体转换数据并传递。简单易推理。一个阶段失败会阻塞其余。

**模式 2：扇出/扇入**
```
           ┌→ 研究者 A →┐
输入 → 拆分 ├→ 研究者 B →├→ 合并 → 输出
           └→ 研究者 C →┘
```
拆分到并行智能体，然后合并结果。适合可分解为独立子任务的任务。

**模式 3：编排者-工作者**
```
           ┌─ 编排者 ─┐
        任务│         │任务
     ┌─────┘         └─────┐
     ▼                     ▼
┌──────────┐         ┌──────────┐
│ 工作者 A │         │ 工作者 B │
└──────────┘         └──────────┘
```
智能编排者决定做什么、委派给工作者、合成结果。

**模式 4：对等群体**
```
  ┌─ A ◄─ 消息 ─→ B ─┐
  │    ┌─ 共享状态 ─┐  │
  └─ C ◄─ 消息 ─→ D ─┘
```
无中心编排者。智能体点对点通信。决策从交互中涌现。更难调试但能扩展到更多智能体。

### 2.5 何时不用多智能体

多智能体增加复杂性。每个智能体间的消息都是潜在的失败点。

**保持单智能体当：**
- 任务能放在一个上下文窗口中（工作数据低于 ~100K 词元）
- 不需要不同阶段的不同系统提示词
- 顺序执行足够快
- 任务简单到拆分增加的开销超过价值

经验法则：如果任务少于 20 次工具调用且能放在 100K 词元内，保持单智能体。

## 3. 从零实现

### 第 1 步：单智能体方法

```python
def single_agent_approach(task):
    """一个巨大的系统提示词 + 一个上下文窗口。"""
    system_prompt = "你是全栈开发者。你必须：1.研究需求 2.写代码 3.审查代码 4.写测试。"
    context_window = []

    research = fake_llm_call(system_prompt, f"研究: {task}")
    context_window.append(research.output)

    code = fake_llm_call(system_prompt, f"根据研究:\n{chr(10).join(context_window)}\n\n为 {task} 写代码")
    context_window.append(code.output)

    review = fake_llm_call(system_prompt, f"审查代码。")
    context_window.append(review.output)

    return {"content": "\n---\n".join(context_window), "tokens": ...}
```

### 第 2 步：多智能体流水线

```python
researcher = create_specialist("researcher", "你是技术研究员。读文档、找模式、总结发现。")
coder = create_specialist("coder", "你是高级 Python 开发者。根据研究笔记写代码。")
reviewer = create_specialist("reviewer", "你是代码审查员。找 bug。具体指出行号。")

def multi_agent_pipeline(task):
    """每个智能体一个上下文窗口。"""
    research = specialist_run(researcher, task)
    code = specialist_run(coder, f"[来自研究者]: {research.content}")
    review = specialist_run(reviewer, f"[来自编码者]: {code.content}")
    return review
```

### 第 3 步：扇出并行

```python
async def multi_agent_fan_out(task):
    """研究者和需求分析并行。"""
    loop = asyncio.get_event_loop()
    research_future = loop.run_in_executor(None, specialist_run, researcher, f"研究 {task}")
    requirements_future = loop.run_in_executor(None, specialist_run, requirements, f"分析 {task} 需求")
    research_result, requirements_result = await asyncio.gather(research_future, requirements_future)
    # 合并后传给编码者
    code_result = specialist_run(coder, f"研究: {research_result.content}\n需求: {requirements_result.content}")
    return code_result
```

## 4. 工业工具

### 4.1 模式选择指南

| 模式 | 延迟 | 成本 | 调试 | 适用 |
|------|------|------|------|------|
| 单智能体 | 最低 | 最低 | 最简单 | 简单任务 |
| 子智能体 | 低 | 中 | 简单 | 专注子任务 |
| 流水线 | 中 | 中 | 中等 | 分阶段工作 |
| 扇出/扇入 | 低（并行） | 中高 | 中等 | 独立子任务 |
| 编排者-工作者 | 中高 | 高 | 困难 | 智能协调 |
| 对等群体 | 变化大 | 最高 | 最困难 | 大规模并行 |

## 5. 工程最佳实践

### 5.1 多智能体设计原则

| 原则 | 说明 |
|------|------|
| 每个智能体一个任务 | 不要混合角色 |
| 清晰的通信契约 | 消息是结构化数据，不是自由文本 |
| 共享状态要最小化 | 智能体间共享的状态越多，调试越难 |
| 先单后多 | 只在单智能体碰到天花板时才拆分 |

## 6. 常见错误

### 错误 1：过早引入多智能体

**现象：** 一个简单的 10 步任务被拆成 5 个智能体。调试比工作本身更耗时。

**修复：** 经验法则——少于 20 次工具调用且 100K 词元以内，保持单智能体。

### 错误 2：智能体间没有清晰的通信契约

**现象：** 研究者返回文本，编码者以自己的方式解析。两个智能体对同一段文本有不同理解。

**修复：** 使用结构化消息格式——JSON schema、类型化工件、显式协议。

### 错误 3：共享过多状态

**现象：** 5 个智能体共享一个大状态对象。一个智能体的修改意外影响另一个。

**修复：** 最小化共享状态。智能体通过消息通信，不共享可变状态。

## 7. 面试考点

### Q1：单智能体天花板的三个表现是什么？（难度：⭐）

**参考答案：**
1. 上下文饱和——到第 30 轮消耗 150K 词元，关键细节丢失
2. 角色混乱——一个提示词试图覆盖多个角色，每样都做得平庸
3. 顺序瓶颈——串行 LLM 调用，无并行

### Q2：四种编排模式各适合什么场景？（难度：⭐⭐）

**参考答案：**
- 流水线：分阶段工作（研究→编码→审查），简单易推理
- 扇出/扇入：可分解为独立子任务，需要并行
- 编排者-工作者：需要智能协调，编排者决定做什么并委派
- 对等群体：无中心，大规模并行，决策涌现，难调试

### Q3：多智能体的复杂度权衡是什么？（难度：⭐⭐⭐）

**参考答案：**
延迟增加（N 个智能体至少 N 次串行 LLM 调用）、成本倍增（每个智能体独立消耗词元）、调试困难（从"读一段对话"变成"跨五个智能体追踪消息"）、有损压缩（A 的完整上下文被摘要成一条消息给 B）。经验法则：少于 20 次工具调用且 100K 词元以内，保持单智能体。

## 📚 小结

单智能体天花板的三个表现：上下文饱和、角色混乱、顺序瓶颈。多智能体通过拆分工作解决——每个智能体一个任务、一个上下文窗口、一个专注提示词。四种编排模式覆盖从简单流水线到对等群体的全谱。但多智能体增加复杂度。经验法则：少于 20 次工具调用且 100K 词元以内，保持单智能体。
