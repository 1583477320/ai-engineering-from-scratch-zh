# 并行/群体/网络化架构

> 与监督者对比：没有中央决策者。智能体读取共享事件总线，异步拾取工作，写回结果。LangGraph 明确支持用于去中心化动态环境的"群体架构"。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，消除编排者瓶颈。权衡是明确的：确定性和可追溯性换可扩展性。群体适合有许多独立子问题的任务；不适合需要单一连贯计划的任务。

## 1. 问题

监督者扩展到少数工作者。但数百个呢？监督者本身成为瓶颈——每个关于谁做什么的决策都通过一个智能体。一个慢规划步骤就阻塞整个系统。

群体架构翻转设计。不是中央规划者分派工作，工作者从共享队列拉取工作。"协调"内嵌在事件总线语义中。没有编排者；系统扩展直到队列不行。

核心洞察：**监督者是为少数工作者设计的；群体是为大量独立子问题设计的。** 选择取决于任务的独立性。

## 2. 概念

### 2.1 形状

```
                ┌──── 共享队列 ────┐
                │                   │
       ┌────────┼────────┐  ◄───────┘
       ▼        ▼        ▼
     工作者   工作者   工作者
      A       B       C
       │        │        │
       └────────┴────────┘
                 │
                 ▼
            结果池
```

没有编排者。每个工作者重复：拉取任务、处理、写结果（可选：入队后续）。

### 2.2 群体何时合适

| 场景 | 为什么合适 | 示例 |
|------|----------|------|
| 大量独立任务 | 任务之间不依赖 | 爬取、转换、分类 |
| 可变时长工作 | 群体自动平衡负载 | 有些 100ms，有些 10s |
| 吞吐量优先于确定性 | 关心总完成时间，不关心严格排序 | 批处理工作 |

### 2.3 群体何时失败

| 场景 | 为什么失败 | 替代方案 |
|------|----------|---------|
| 有序工作流 | 步骤 3 需要步骤 2 的输出 | 流水线（阶段 16 · 01） |
| 全局规划任务 | 群体产生独立事实而非连贯报告 | 监督者（阶段 16 · 05） |
| 调试 | 没有中央日志，重现 bug 代价高 | 需要每任务追踪 ID |

### 2.4 Matrix 框架（arXiv:2511.21686）

Matrix 将群体推到自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调者。容错来自消息持久性。可扩展性是消息代理的问题，不是系统的问题。

编程模型：**"这个智能体订阅什么消息主题？"** 而不是"编排者下一个选谁？"。这使系统看起来像发布/订阅事件网格。

### 2.5 饥饿和热点

**饥饿**：如果所有工作者拉取最快可用的任务，长运行任务永远不会被拾取直到它们是唯一剩下的。经典的队列饥饿。

**热点**：一个工作者处理了大部分任务，其他工作者空闲。负载不均衡。

| 缓解 | 说明 |
|------|------|
| 优先级队列 + 老化 | 等待时间越长优先级越高 |
| 工作者专业化 | 有些工作者只接长任务 |
| 背压 | 限制进入队列的快速任务数量 |
| 每任务追踪 ID | 调试需要 |

### 2.6 Anthropic 的选择

Anthropic 在其 Research 系统中明确选择了监督者而非群体（阶段 16 · 05）。原因：研究任务需要全局规划——将子问题分解为连贯的子问题集合。群体产生独立事实而非连贯报告。群体适用于大量独立任务；监督者适用于需要全局规划的任务。

## 3. 从零实现

### 第 1 步：定义任务和工作者

```python
import queue
import threading
import time
from dataclasses import dataclass

@dataclass
class Task:
    task_id: int
    duration: float
    pre_assigned: int  # 固定分配基线用

def fake_work(task):
    time.sleep(task.duration)
    return f"task-{task.task_id}-done"
```

### 第 2 步：实现三种调度策略

```python
def run_sequential(tasks):
    """顺序执行：一个工作者处理所有任务。"""
    t0 = time.time()
    for t in tasks:
        fake_work(t)
    return time.time() - t0

def run_fixed_assignment(tasks, n_workers):
    """固定分配：每个任务预先分配给一个工作者。"""
    per_worker = {i: [] for i in range(n_workers)}
    for t in tasks:
        per_worker[t.pre_assigned].append(t)
    counts = {i: 0 for i in range(n_workers)}
    def worker(wid):
        for t in per_worker[wid]:
            fake_work(t)
            counts[wid] += 1
    t0 = time.time()
    threads = [threading.Thread(target=worker, args=(i,)) for i in range(n_workers)]
    for th in threads: th.start()
    for th in threads: th.join()
    return time.time() - t0, counts

def run_swarm(tasks, n_workers):
    """群体：工作者从共享队列拉取。"""
    q = queue.Queue()
    for t in tasks:
        q.put(t)
    counts = {i: 0 for i in range(n_workers)}
    lock = threading.Lock()
    def worker(wid):
        while True:
            try:
                task = q.get_nowait()
            except queue.Empty:
                return
            fake_work(task)
            with lock:
                counts[wid] += 1
            q.task_done()
    t0 = time.time()
    threads = [threading.Thread(target=worker, args=(i,)) for i in range(n_workers)]
    for th in threads: th.start()
    for th in threads: th.join()
    return time.time() - t0, counts
```

### 第 3 步：运行对比

```python
def main():
    # 8 个任务：一半慢（0.4s），一半快（0.1s）
    tasks = []
    for i in range(8):
        is_slow = i < 4
        tasks.append(Task(i, 0.4 if is_slow else 0.1, 0 if is_slow else (i - 3) % 4))
    n_workers = 4

    seq_time = run_sequential(tasks)
    fixed_time = run_fixed_assignment(tasks, n_workers)
    swarm_time, swarm_counts = run_swarm(tasks, n_workers)

    print(f"顺序: {seq_time:.2f}s")
    print(f"固定分配: {fixed_time:.2f}s")
    print(f"群体: {swarm_time:.2f}s")
    print(f"群体加速 vs 顺序: {seq_time / swarm_time:.2f}x")
    print(f"群体加速 vs 固定:  {fixed_time / swarm_time:.2f}x")
```

## 4. 工业工具

### 4.1 群体 vs 监督者选择

| 维度 | 群体 | 监督者 |
|------|------|--------|
| 任务独立性 | 高（独立子任务） | 低（需要全局规划） |
| 可变时长 | 自动平衡负载 | 需要预测时长 |
| 确定性 | 低 | 高 |
| 可调试性 | 困难 | 容易 |
| 可扩展性 | 高（到数千智能体） | 有限（监督者瓶颈） |

### 4.2 生产部署检查清单

| 检查项 | 说明 |
|--------|------|
| 优先级队列 + 老化 | 防止长任务饥饿 |
| 工作者幂等性 | 任务可能被多次拉取（工作者崩溃时） |
| 持久化队列 | Kafka/Redis Streams/数据库——`queue.Queue` 仅内存 |
| 每任务可观测性 | 每个任务有追踪 ID |
| 背压 | 队列增长快于工作者消耗时，减速生产者 |

## 5. 工程最佳实践

### 5.1 群体设计原则

| 原则 | 说明 |
|------|------|
| 优先级队列 + 老化 | 防止长任务饥饿 |
| 工作者幂等性 | 崩溃恢复时任务可能被多次拉取 |
| 持久化队列 | 生产用 Kafka/Redis，不用内存队列 |
| 每任务追踪 ID | 调试需要 |

### 5.2 中文场景特别建议

- **群体在中文 NLP 批处理中有效**——中文文本分类、情感分析可以并行运行
- **背压机制在中文云环境中同样重要**——阿里云、腾讯云的队列服务支持背压配置
- **工作者幂等性在分布式中文环境中更关键**——网络波动可能导致任务重复执行
- **持久化队列在中文云环境中可用**——阿里云 RocketMQ、腾讯云 CMQ 都支持

## 6. 常见错误

### 错误 1：长任务饥饿

**现象：** 所有工作者拉取最快任务，长运行任务永远不被拾取直到最后。

**修复：** 优先级队列 + 显式老化（等待时间越长优先级越高）+ 工作者专业化（有些只接长任务）。

### 错误 2：无追踪 ID 的调试

**现象：** 任务失败但无法追溯——没有追踪 ID，不知道是哪个工作者处理的。

**修复：** 每个任务有追踪 ID；每个工作者记录开始/结束。

### 错误 3：内存队列在生产中

**现象：** 用 `queue.Queue` 运行生产负载。工作者崩溃时任务丢失。

**修复：** 生产用 Kafka、Redis Streams 或数据库队列。`queue.Queue` 仅内存。

## 7. 面试考点

### Q1：群体架构与监督者架构的区别是什么？（难度：⭐）

**参考答案：**
监督者有中央决策者分派工作；群体中工作者从共享队列拉取工作，没有中央编排者。群体可扩展到数千智能体（队列是瓶颈），监督者在少数工作者时就遇到瓶颈。

### Q2：群体的三种失败模式是什么？（难度：⭐⭐）

**参考答案：**
1. **饥饿**——所有工作者拉取最快任务，长运行任务永不被拾取
2. **热点**——一个工作者处理了大部分任务，负载不平衡
3. **调试困难**——没有中央日志和异步工作，重现 bug 代价高

缓解：优先级队列 + 老化；工作者专业化；每任务追踪 ID。

### Q3：Matrix 框架的核心贡献是什么？（难度：⭐⭐⭐）

**参考答案：**
将控制流和数据流都表示为分布式队列上的序列化消息。没有中央协调者——可扩展性是消息代理的问题。

编程模型："这个智能体订阅什么消息主题？"而不是"编排者下一个选谁？"。容错来自消息持久性。这消除了编排者瓶颈。

### Q4：为什么 Anthropic 在生产中选择监督者而非群体？（难度：⭐⭐⭐）

**参考答案：**
研究任务需要全局规划——将子问题分解为连贯的子问题集合。群体产生独立事实而非连贯报告。监督者通过领智能体的计划和合成步骤来保证全局连贯性。

群体适用于大量独立任务（爬取、转换）；监督者适用于需要全局规划的任务（研究、分析）。选择取决于任务的独立性。

## 🔑 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 群体架构 | "去中心化智能体" | 工作者从共享队列拉取；无中央编排者 |
| 事件总线 | "智能体订阅主题" | 按类型或内容将任务路由给工作者的消息代理 |
| 饥饿 | "任务永不运行" | 低优先级任务因高优先级工作持续到达而永不被拾取 |
| 热点 | "一个工作者被淹没" | 一个工作者处理了大部分任务的负载不均衡 |
| 背压 | "减速生产者" | 队列满时通知上游停止生产的机制 |
| 幂等工作者 | "可安全重跑" | 任务处理两次产生相同结果。工作者崩溃时需要 |
| Matrix 框架 | "全消息传递群体" | 控制流和数据流都是分布式队列上的序列化消息 |

## 📚 小结

群体架构：工作者从共享队列拉取工作，没有中央编排者。适合大量独立任务、可变时长工作、高吞吐量。三种失败：饥饿、热点、调试困难。Matrix 将控制流和数据流都表示为分布式消息。权衡：确定性和可追溯性换可扩展性。群体不适合需要单一连贯计划的任务。Anthropic 在生产中选择监督者而非群体——研究需要全局规划。
