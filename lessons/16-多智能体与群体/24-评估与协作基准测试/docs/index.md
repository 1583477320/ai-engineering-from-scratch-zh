## 1. 问题

当一篇论文声称"我们的多智能体系统更好"时，问题是：比什么好了？在什么任务上？用什么指标衡量的？

2023-2024 年的多智能体评估是一片混乱——每个人挑自己的指标、自己的基线、自己的任务集。2025-2026 年的基准测试带来了结构。没有共享的基准测试，你就无法有意义地比较两个多智能体系统。更糟糕的是，没有保持不发布的保留基准测试，前沿模型的训练数据可能会污染基准测试——SWE-bench Verified 在中 2025 年就已经被部分污染，分数被严重膨胀。

**SWE-bench Pro 就是为此设计的无污染的现实检查。** 前沿模型在 Pro 上的分数约 23%，在 Verified 上却超过 70%——这个差距就是污染信号。

## 2. 概念

### 2.1 MultiAgentBench / MARBLE——ACL 2025

arXiv:2503.01935。评估四种协作拓扑（星型、链型、树型、图型）在研究、编码和规划任务上的表现。使用里程碑式的 KPI——不仅衡量最终成功，还衡量中间进展。

实测结果：

- **图型拓扑**在研究场景中最佳——支持任意对任意的评审
- **链型拓扑**在逐步精化编码中最佳
- **星型拓扑**在快速事实整合中最佳
- **协作税**在图型拓扑超过约 4 个智能体后出现
- **认知规划**在所有拓扑上增加约 3% 的里程碑达成率

适用场景：你想对协作拓扑做苹果到苹果的对比。

### 2.2 COMMA——多模态非对称信息

涵盖智能体拥有不同观察模态且必须在不全信息共享的情况下协调的任务。报道的结果令人不安：**包括 GPT-4o 在内的前沿模型在 COMMA 的智能体-智能体协作上难以超越随机基线。** 信号是：多模态协调在 LLM 领域训练不足、评估不足。单模态协作尚可；多模态协调直接崩溃。

适用场景：你的系统涉及多模态或非对称信息协调。COMMA 的零结果是行动前的警告。

### 2.3 MedAgentBoard——领域压力测试

arXiv:2505.12371。四个医疗任务类别：诊断、治疗方案制定、报告生成、患者沟通。比较多智能体 vs 单 LLM vs 传统规则系统。

发现：**多智能体在大多数类别上并不优于单 LLM。** 多智能体的优势很窄——当子任务可以清晰分离时（诊断+治疗），任务分解有帮助；当协作开销超过专业化收益时（报告生成），反而有害。

适用场景：你的领域有清晰的单 LLM 基线。如果 MedAgentBoard 的教训有普遍性，很多被提议的多智能体系统都是过度工程。

### 2.4 AgentArch——企业架构

arXiv:2509.10769。企业环境中的工具使用、记忆和编排分层评估。基准测试隔离了每个层的贡献：添加工具有多大帮助？添加记忆呢？添加多智能体编排呢？

适用场景：你在设计企业智能体技术栈，需要为每一层提供合理性论证。AgentArch 帮助你避免购买你无法衡量价值的特性。

### 2.5 SWE-bench Pro——无污染的现实检查

arXiv:2509.16941。1865 个问题，覆盖 41 个仓库，涵盖业务应用、B2B 服务和开发者工具。设计目标是**不被污染**——使用比训练数据截止日期更晚的问题。前沿模型在 Pro 上约 23%，在 Verified 上超过 70%——这个差距就是污染信号。

2026 年 4 月的得分：
- Claude Opus 4.7 在 Pro 上：**64.3%**（使用显式智能体团队协作；尚未发布 Anthropic 原始论文——视为初步结果）
- Verdent（智能体脚手架）在 Verified 上：**76.1% pass@1**
- 无智能体脚手架的前沿模型在 Pro 上：约 23-35%

**教训：** "我们超过了 SWE-bench Verified" 不再有能力证明力。Pro 是当前的看门测试。

### 2.6 基准测试声明审查清单

当有人声称多智能体结果时，问六个问题：

1. **哪个基准测试，哪个数据集？** SWE-bench Verified vs Pro 差异极大。错误数据集上的数字毫无价值
2. **污染检查。** 基准测试是在模型的训练截止日期之后发布的吗？如果不是，保持警惕
3. **基线对比。** vs 单 LLM 基线、vs 随机基线、vs 之前的多智能体工作。不是"vs 未调优版的同一系统"
4. **统计显著性。** 试验次数、p 值、置信区间。前沿模型方差大；单次运行误导
5. **任务多样性。** 一个任务还是多个？泛化能力对生产至关重要
6. **成本披露。** 每个任务的词元数、墙钟时间。90% 的解决方案付出 20 倍的代价是一个商业决策，不是能力声明

### 2.7 基准测试共同缺失的维度

所有当前基准测试都做得不好的方面：

- **长时程协作。** 数天的墙钟交互。所有当前基准测试都跑短任务
- **对抗鲁棒性。** 当一个智能体被恶意攻击或妥协时会发生什么？
- **部署漂移。** 基准测试是静态的；生产分布会迁移
- **成本归一化性能。** 大多数基准测试报告原始准确率，而不是每美元的准确率

建立针对你实际关心的维度的内部基准测试通常是正确的做法。

## 3. 从零实现

### 第 1 步：里程碑 KPI 计算

```python
class MilestoneTracker:
    """里程碑式 KPI——追踪部分进展而非仅最终成功。"""

    def __init__(self, milestones: list[str]):
        self.milestones = milestones
        self.completed = set()
        self.attempts = {}

    def mark_complete(self, milestone: str):
        """标记里程碑为完成。"""
        self.completed.add(milestone)
        self.attempts[milestone] = self.attempts.get(milestone, 0) + 1

    def achievement_rate(self) -> float:
        """里程碑达成率。"""
        return len(self.completed) / len(self.milestones)

    def milestone_scores(self) -> dict:
        """每个里程碑的完成情况。"""
        return {m: m in self.completed for m in self.milestones}
```

### 第 2 步：基准测试声明审查

```python
class BenchmarkClaimChecker:
    """基准测试声明审查——应用六问审查清单。"""

    def __init__(self, claim: str):
        self.claim = claim
        self.checks = {
            "benchmark_specified": False,
            "contamination_checked": False,
            "baseline_comparison": False,
            "statistical_significance": False,
            "task_diversity": False,
            "cost_disclosed": False,
        }

    def grade(self) -> str:
        """对声明打分。"""
        score = sum(self.checks.values()) / len(self.checks)
        if score >= 0.83:
            return "A — 可信"
        elif score >= 0.5:
            return "B — 部分可信，需要更多验证"
        else:
            return "C — 证据不足，保持怀疑"

    def report(self) -> dict:
        """输出审查报告。"""
        return {
            "checks": self.checks,
            "grade": self.grade(),
        }


if __name__ == "__main__":
    # 示例：对一个假设的声明做审查
    checker = BenchmarkClaimChecker("我们的系统在 SWE-bench Verified 上达到了 75%")
    checker.checks = {
        "benchmark_specified": True,
        "contamination_checked": False,  # Verified 已被污染
        "baseline_comparison": False,
        "statistical_significance": False,
        "task_diversity": True,
        "cost_disclosed": False,
    }
    print(f"评分: {checker.grade()}")
```

## 5. 工程最佳实践

### 5.1 建立你自己的内部基准

公共基准测试提供参考但不替代。建立反映你实际生产分布的内部基准测试。

**必须在内部基准中包含的内容：**
- 随机基线——如果你在协作任务上不能大幅超越随机，任务可能设计不当
- 成本 + 准确率并列报告——运营团队需要两者
- 按领域分层——不要平均一个数字，按场景看性能分布

### 5.2 引用基准测试的规则

- 用 SWE-bench Pro，不用 Verified（Verified 已被污染）
- 加基线对比（随机基线 + 单 LLM 基线 + 最优可用系统）
- 加成本披露（词元/任务、墙钟时间）
- N ≥ 5 次运行，报告均值和方差

### 5.3 每季度重建基准

生产分布会迁移；过时的基准测试误导判断。
- 从生产中抽样更新内部基准
- 跟踪基准分数与生产指标之间的相关性
- 当相关性下降时重建基准

### 5.4 中文场景特别建议

- **中文编码基准测试不足。** SWE-bench Pro 以英文仓库为主。中文多智能体编码系统缺乏评估标准——建议在内部基准中加入中文软件工程任务
- **中文多模态基准测试更少。** COMMA 的多模态评估以英文场景为主。中文医疗/教育领域的多智能体系统需要自建评估
- **国内模型的 SWE-bench 分数需单独报告。** 中文厂商的模型（通义、文心、DeepSeek）在 SWE-bench 上的公开分数不可直接对比——测试环境和评估方法不同

## 6. 常见错误

### 错误 1：只用 SWE-bench Verified

**现象：** "我们的系统在 SWE-bench Verified 上达到了 80%！"——但 Pro 上只有 20%。

**原因：** Verified 已被训练数据污染。前沿模型在训练时可能见过测试集。

**修复：** 切换到 SWE-bench Pro，或在报告中同时标注 Verified 和 Pro 分数。

### 错误 2：没有随机基线

**现象：** 论文声称"多智能体方法比单 LLM 好 10%"——但在某些任务上，随机基线比两者都好。

**原因：** 任务本身可能太难或太模糊，任何方法都无法有效解决。

**修复：** 始终包含随机基线。如果你不能大幅超越随机，重新检查任务的可行性。

### 错误 3：单次运行报结果

**现象：** "准确率 85%"。3 次重跑后发现分别是 65%、85%、45%。

**原因：** LLM 输出方差大，单次运行不可靠。

**修复：** 至少 5 次运行，报告均值和置信区间。

## 7. 面试考点

### Q1：SWE-bench Verified 和 Pro 之间的核心区别是什么？为什么这个区别如此重要？（难度：⭐⭐）

**参考答案：**
Verified 被训练数据污染——前沿模型可能在训练时已经见过测试集，导致分数虚高（70%+）。Pro 使用比模型训练截止日期更晚的问题，确保模型不可能在训练时见过测试集——前沿模型在 Pro 上约 23%。这个差距就是量化了的污染信号。目前，用 Verified 做基准测试声明已经没有说服力了。

### Q2：MARBLE 发现"协作税"在图型拓扑中随着智能体数量增长而出现——为什么会这样？这对系统设计有什么影响？（难度：⭐⭐⭐）

**参考答案：**
图型拓扑允许任意两个智能体之间直接评审和通信。智能体数量从 4 增加到 8 时，通信链路从 6 条增加到 28 条（O(n²)）。每条链路都可能产生评审请求、反馈、修改——智能体花费在沟通上的时间超过了实际工作的时间。影响：如果你需要超过 4 个智能体，不要使用纯图型拓扑——考虑分层（组内密集 + 组间稀疏）或混合拓扑。

### Q3：你的团队内部完成了自己的多智能体系统，想用一个公共基准来证明效果。你的选择步骤是什么？（难度：⭐⭐）

**参考答案：**
第一步：确定与你的任务领域最相关的基准测试（编码→SWE-bench Pro，研究→MARBLE，多模态→COMMA）。第二步：在报告中同时包含随机基线和单 LLM 基线——不是只和自己比较。第三步：运行至少 5 次，报告均值和方差。第四步：提供成本数据（词元/任务 + 墙钟时间）。第五步：在论文或报告中公开承认基准测试的局限性——没有基准测试能完美模拟生产环境。

## 📚 小结

2025-2026 年的五个基准测试（MARBLE、COMMA、MedAgentBoard、AgentArch、SWE-bench Pro）为多智能体评估带来了结构化——但在使用它们时必须保持批判性。SWE-bench Verified 被污染、COMMA 的前沿模型无法超越随机基线、MedAgentBoard 发现多智能体不优于单 LLM——这些信号提醒你：基准测试只是工具，不是答案。建立你自己的内部基准、始终包含基线对比、报告成本 > 原始分数、理解每个基准测试具体衡量什么。课程：一个成熟的领域是有基准测试的；一个成熟的多智能体工程师是会批判性阅读的。
