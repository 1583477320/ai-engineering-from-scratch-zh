# 影子流量、金丝雀发布与渐进式部署

> LLM 发布结合了软件部署中最困难的部分：没有单元测试、弥散的失败模式、延迟信号。流程是 (1) 影子模式——将生产请求复制到候选模型，记录、对比，零用户影响；(2) 金丝雀发布——渐进式流量转移 10%→25%→50%→75%→100%，每步有门控；(3) A/B 测试——在稳定性确认后比较明确的替代方案。非确定性是不可约的——相同输入的准确率变化可达 15%。

**类型：** 概念课
**语言：** Python
**前置知识：** 阶段 17 · 13（可观测性）、阶段 17 · 21（A/B 测试）
**预计时间：** ~60 分钟
**所处阶段：** Tier 3

## 1. 问题

你发布了新模型。离线评估显示 3% 准确率提升。你在生产中打开它。24 小时内，成本上升 40%，用户踩票上升 8%，三个客户工单报告"奇怪的答案"。你回滚。重新部署需要 3 小时。你的周末毁了。

这些每一个都是可以避免的。影子模式会在任何用户看到之前捕获 40% 的成本飙升。金丝雀会在踩票变动时停在 10%。策略标志回滚只需 30 秒。这个纪律填补了"离线评估看起来很好"和"真实用户满意"之间的鸿沟。

## 2. 概念

### 2.1 影子模式

候选模型接收与生产相同的请求；输出被记录但不返回给用户。零用户影响。记录：

- 输出内容（与生产对比的 diff）
- 词元数（成本差异）
- 延迟
- 拒绝和错误

能捕获：成本爆炸、长度回归、明显的拒绝变化、硬错误。不能捕获：用户会感知的质量差异。影子是烟雾测试，不是质量测试。

### 2.2 金丝雀发布

渐进式流量转移，带门控。典型进展：1%→10%→25%→50%→75%→100%。每步检查五个指标：

1. **延迟百分位** — P50、P95、P99。违约：金丝雀 P99 > 1.5 倍基线
2. **每请求成本** — 混合 $。违约：>20% 高于基线
3. **错误/拒绝率** — 5xx 加显式拒绝。违约：2 倍基线
4. **输出长度分布** — 均值 + P99。违约：分布性偏移
5. **用户反馈率** — 踩票/工单。违约：1.5 倍基线

### 2.3 非确定性是新的方差

相同输入产生非相同输出。原因：

- GPU FP 非结合性（浮点归约顺序随批次变化）
- 批大小方差（相同提示词在 128 批次 vs 16 批次中）
- 采样（temperature > 0）

测量：相同评估集上的运行间准确率变化最高 15%。发布中的"稳定"意味着指标在预期方差内，而非与基线相同。门控阈值要设在噪声基线之上。

### 2.4 成本是变量

一个好 20% 的模型每次调用可能贵 3 倍。每请求成本是五个门控之一。发布一个"更好"但打破经济模型的模型是回滚案例。

### 2.5 回滚是武器

- 策略标志（Feature Flag 系统）：在配置中切换百分比；秒级
- 模型锁定（注册表摘要）：锁定的模型不会自动升级
- 回滚 = 恢复标志 + 设置锁定摘要为之前的版本。秒级，不是小时级

如果你的栈需要重新部署来回滚，先修好这个再发布。

### 2.6 工具

- **Argo Rollouts / Flagger** — Kubernetes 渐进式交付控制器
- **Istio 加权路由** — 服务网格级流量分割
- **KServe / Seldon Core** — 带内置金丝雀的模型服务
- **Feature flags** — LaunchDarkly、Flagsmith、Unleash。策略级切换，无需重新部署

## 3. 从零实现

### 第 1 步：金丝雀发布模拟器

```python
import random


def simulate_canary(stage_pct, n_requests, baseline_error_rate=0.05,
                    canary_error_rate=0.05, cost_multiplier=1.0):
    """模拟金丝雀发布的一个阶段。"""
    canary_requests = int(n_requests * stage_pct)
    baseline_requests = n_requests - canary_requests

    canary_errors = int(canary_requests * canary_error_rate)
    baseline_errors = int(baseline_requests * baseline_error_rate)

    total_errors = canary_errors + baseline_errors
    error_rate = total_errors / n_requests

    canary_cost = canary_requests * 0.01 * cost_multiplier
    total_cost = baseline_requests * 0.01 + canary_cost

    return {
        "stage": stage_pct,
        "error_rate": error_rate,
        "total_cost": total_cost,
        "gate_passed": error_rate < 0.08,  # 门控：错误率 < 8%
    }


# 模拟金丝雀进展
print("=== 金丝雀发布模拟 ===\n")
for stage in [0.01, 0.10, 0.25, 0.50, 0.75, 1.0]:
    r = simulate_canary(stage, 1000, cost_multiplier=1.3)
    status = "✓ 通过" if r["gate_passed"] else "✗ 失败"
    print(f"  {r['stage']:>4.0%}  错误率={r['error_rate']:.1%}  "
          f"成本=${r['total_cost']:.2f}  {status}")
```

## 5. 工程最佳实践

### 5.1 影子→金丝雀→A/B 的三步走

先影子（零影响对比）→ 金丝雀（渐进式流量转移）→ A/B（稳定性确认后的对比）。不要跳步。

### 5.2 回滚必须秒级

如果回滚需要重新部署，先修好再发布。回滚 = 策略标志切换 + 模型锁定回退。

### 5.3 中文场景特别建议

- **金丝雀指标的中文语境。** 用户反馈率在中文场景中可能表现为"点赞/踩"和"客服工单"——中文用户的投诉渠道可能不同
- **成本门控。** 国内 LLM API 的定价差异大——从免费开源到 $10/M 不等。金丝雀的成本门控需要根据具体提供商调整
- **非确定性在中文 LLM 上更高。** 中文 LLM 的 temperature 方差可能比英文更大——门控阈值要更宽松

## 6. 常见错误

### 错误 1：跳过影子模式直接金丝雀

**现象：** 金丝雀 10% 流量时发现成本飙升 40%。用户已经开始投诉。

**原因：** 没有先做影子模式对比——成本飙升本应在零影响下被发现。

**修复：** 永远先影子（至少 24 小时），确认成本和长度无异常后再金丝雀。

### 错误 2：回滚需要重新部署

**现象：** 金丝雀发现问题后回滚需要 3 小时——因为回滚意味着重新部署新版本的代码。

**原因：** 模型版本和部署耦合。没有使用策略标志或模型锁定。

**修复：** 将模型版本和部署解耦——策略标志切换+模型注册表回退=秒级回滚。

## 7. 面试考点

### Q1：影子模式、金丝雀和 A/B 的区别是什么？（难度：⭐⭐）

**参考答案：**
影子模式：候选模型接收相同请求，输出记录但不返回给用户——零影响对比。捕获成本、长度、拒绝等硬指标。金丝雀：渐进式流量转移（1%→10%→...→100%），每步检查五个指标门控。捕获延迟、成本、错误、用户反馈。A/B：在稳定性确认后，比较两个明确的替代方案（如不同提示词策略）。三步走：先影子（零影响），再金丝雀（渐进），最后 A/B（稳定性确认后对比）。

### Q2：为什么 LLM 的非确定性使得金丝雀门控更难？（难度：⭐⭐⭐）

**参考答案：**
传统软件的金丝雀门控基于确定性输出——新版本的错误率如果和基线不同，就能检测到。但 LLM 的非确定性意味着相同输入的准确率变化可达 15%——这是噪声，不是故障。如果门控阈值设在噪声基线以下（如错误率 < 5%），会频繁误报。解决方案：门控阈值必须设在噪声基线之上（如错误率 < 8%），并用统计显著性检验（而非单次快照）来判断是否真的有问题。

## 📚 小结

LLM 发布有三个阶段：影子（零影响对比）→ 金丝雀（渐进式流量+门控）→ A/B（稳定性确认后对比）。五个金丝雀门控：延迟、成本、错误率、输出长度分布、用户反馈。非确定性（最高 15%）要求门控阈值设在噪声基线之上。回滚必须秒级——策略标志切换+模型锁定回退，而非重新部署。发布纪律填补了"离线评估看起来很好"和"真实用户满意"之间的鸿沟。
