# LLM 生产的混沌工程

> LLM 的混沌工程在 2026 年是一门独立学科。在生产中运行实验的前提：定义 SLI/SLO、追踪+指标+日志可观测性、自动回滚、手册、值班。架构有四个平面：控制（实验调度器）、目标（服务、基础设施、数据存储）、安全（守卫+终止+流量过滤）、可观测性（指标+追踪+日志）。守卫是强制性的：烧毁率告警在每日错误预算烧毁超过预期 2 倍时暂停实验。节奏：每周小金丝雀+SLO 回顾；每月游戏日+事后分析；每季度跨团队韧性审计。LLM 特定实验：内存过载、网络故障、提供商宕机、畸形提示词、KV 缓存驱逐风暴。

## 1. 问题

传统技术栈的混沌测试已经成熟。LLM 技术栈增加了新的失败模式。一个 4K 词元的提示词中的一个中毒字符让分词器停顿 12 秒。上游提供商 429；你的网关重试；你的服务因重试放大的并发而 OOM。突发负载下的 KV 缓存驱逐风暴导致 re-prefill 级联饱和计算。

这些在单元测试中都不会出现。混沌工程是让你在用户之前发现它们的方式。

## 2. 概念

### 2.1 前提条件

不要在没有以下条件的情况下在生产中运行混沌：

1. **SLI/SLO** — 定义的服务级指标和目标
2. **可观测性** — 追踪、指标、日志，连接到仪表盘
3. **自动回滚** — 第 17 · 20 课的策略标志回滚
4. **手册** — 结构化的，第 17 · 23 课
5. **值班** — 有人响应

缺少任何一个，混沌就变成了真实事故。

### 2.2 四个平面 + 反馈循环

**控制平面** — 实验调度器（Litmus 工作流、Chaos Mesh 计划、Harness UI）

**目标平面** — 服务、pod、节点、负载均衡器、数据存储

**安全平面** — 终止开关、抑制窗口、爆炸半径限制、错误预算门控

**可观测性平面** — 正常指标+追踪 ID 关联，区分混沌引发的和自然的故障

**反馈循环** — 发现反馈到 SLO 调整、手册更新、代码修复

### 2.3 守卫是强制性的

- **烧毁率告警：** 如果每日错误预算烧毁超过预期 2 倍，暂停实验
- **抑制窗口：** 在爆炸半径内静音非实验告警
- **追踪 ID 关联：** 所有实验引发的错误携带标签，值班可去重

### 2.4 五个 LLM 特定实验

1. **内存过载** — 通过高并发发送长上下文请求，强制 KV 缓存抢占风暴。观察：服务是优雅降级还是崩溃？
2. **网络故障** — 切断推理网关和提供商之间的连接。观察：故障切换是否在 SLA 内启动？
3. **提供商宕机模拟** — OpenAI 100% 429。观察：路由是否切换到 Anthropic？
4. **畸形提示词** — 注入让分词器停顿的载荷（深度嵌套 Unicode、巨大 UTF-8 码点）。观察：单个请求是否锁死 worker？
5. **KV 驱逐风暴** — 通过饱和 vLLM 块预算强制驱逐。观察：LMCache 是否恢复还是服务退化？

### 2.5 节奏

- **每周** — 暂存环境小金丝雀实验，可能 5% 生产
- **每月** — 特定场景的计划游戏日；跨团队参与；事后分析
- **每季度** — 跨团队韧性审计；依赖图更新

## 3. 从零实现

### 第 1 步：混沌实验模拟器

```python
def simulate_chaos_experiment(experiment_name, failure_rate,
                              recovery_time, burn_budget):
    """模拟混沌实验的安全检查。"""
    # 烧毁率检查：每分钟消耗的错误预算
    burn_rate = failure_rate * 60  # 每分钟的错误数
    expected_burn = 0.5  # 预期每分钟 0.5 个错误
    is_burn_breach = burn_rate > expected_burn * 2

    # 模拟恢复
    time.sleep(recovery_time)  # 简化

    return {
        "experiment": experiment_name,
        "failure_rate": failure_rate,
        "burn_rate": burn_rate,
        "burn_breach": is_burn_breach,
        "action": "暂停实验" if is_burn_breach else "继续",
    }


if __name__ == "__main__":
    experiments = [
        ("KV 驱逐风暴", 0.3, 120),
        ("提供商宕机", 0.05, 30),
        ("畸形提示词", 0.02, 5),
    ]

    for name, rate, recovery in experiments:
        result = simulate_chaos_experiment(name, rate, recovery, 0.5)
        print(f"{name:15s}  错误率={result['failure_rate']:.1%}  "
              f"烧毁率={result['burn_rate']:.1f}/min  {result['action']}")
```

## 5. 工程最佳实践

### 5.1 从最小实验开始

第一个实验：在稳态流量下一个 decode replica 的 pod kill。观察重路由和恢复。如果这有效且安全，再进入网络混沌。

第一个 LLM 特定实验：注入一个提供商 429 持续 5 分钟。观察故障切换。大多数团队会发现他们的故障切换没有被完全测试。

### 5.2 守卫不可选

烧毁率告警、抑制窗口、追踪 ID 关联——三者缺一不可。没有守卫的混沌工程不是混沌——是随机破坏。

### 5.3 中文场景特别建议

- **国内混沌工程工具。** LitmusChaos 和 Chaos Mesh 都是开源的，可以在国内 K8s 集群上部署。阿里云也有自己的混沌工程产品（ChaosBlade）
- **国内 LLM 提供商的容错。** 国内 LLM API 的限流策略和错误码与 OpenAI/Anthropic 不同——混沌实验需要适配
- **国内网络环境的混沌。** 国内云厂商的内网延迟和丢包率与海外不同——网络故障模拟需要调整参数

## 6. 常见错误

### 错误 1：没有前提条件就运行混沌

**现象：** 混沌实验导致生产事故——没有人知道发生了什么，没有人响应。

**原因：** 没有 SLI/SLO、没有可观测性、没有值班。混沌变成了真实事故。

**修复：** 五个前提条件缺一不可。先修好可观测性和值班，再运行混沌。

### 错误 2：没有烧毁率门控

**现象：** 混沌实验的错误率飙升，但没有自动暂停——错误预算在 10 分钟内烧完了。

**原因：** 没有设置烧毁率告警。

**修复：** 设置烧毁率告警——每日错误预算烧毁超过预期 2 倍时自动暂停实验。
