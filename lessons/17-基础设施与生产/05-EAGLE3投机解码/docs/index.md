# EAGLE-3 投机解码在生产中的实践

> 投机解码将一个快速草稿模型与目标模型配对。草稿提出 K 个词元，目标在一次前向传播中验证——被接受的词元相当于免费的。2026 年 EAGLE-3 是生产级变体——它在目标模型的隐藏状态上训练草稿头，而不是在原始词元上，将接受率 α 推到 0.6-0.8 的区间。正确的问题不是"草稿有多快"，而是"在我的流量上 α 是多少？"。α 低于约 0.55 时，投机解码在高并发下是净负面的。

## 1. 问题

解码是内存带宽密集型的。在运行 Llama 3.3 70B FP8 的 H100 上，每个解码词元读取约 140GB/s 的权重并产出一个词元。GPU 计算在解码时几乎空闲——瓶颈是 HBM 带宽，不是矩阵乘法吞吐。

投机解码利用了这个差距。用一个廉价的草稿模型生成 K 个候选词元，然后让目标模型在一次前向传播中验证所有 K 个。被验证的词元本质上是免费的——它们被分摊到目标模型本来就要做的一次 K-词元批量前向传播中。

经典 draft model 方法使用同系列的小模型（如 Llama 3.2 1B 为 Llama 3.3 70B 草稿）。但接受率 α 只有 0.3-0.5——因为小模型分布与目标分布差异太大。EAGLE → EAGLE-2 → EAGLE-3 在目标模型的隐藏状态上训练轻量草稿头，α 提升到 0.6-0.8。

**关键陷阱：** EAGLE-3 在 vLLM 2026 中是 opt-in。不设 `speculative_config` 就没有加速。很多团队不测量就打开——结果 P99 尾部延迟反而变差了。

## 2. 概念

### 2.1 投机解码真正买到的是什么

不使用投机解码时，每个词元的成本是一次目标前向传播。使用投机解码（草稿长度 K，接受率 α）时，每次目标前向传播期望产出 `1 + K × α` 个词元。加速比 = `(1 + K × α) / (1 + ε)`，其中 ε 是草稿+验证的开销。K=5、α=0.7 时：`(1 + 5×0.7) / (1 + 0.1) = 4.1x`。实际数字通常在 2-3x，因为生产流量中 α 很难达到 0.7，且在高并发下 ε 增长。

### 2.2 为什么 α 是唯一重要的指标

被拒绝的词元不会消失——它们强制目标模型做第二次前向传播来处理第一个被拒绝的词元。在 α 降到 0.4 的工作负载上，你同时付出了草稿开销+验证失败+重滚。在高并发（如 256 并发）下，decode 批次已经足够大，"仅目标"和"带验证"之间的内存带宽差距变小。**α 低于约 0.45-0.55 时，投机解码净负面。**

α 因工作负载而异。在 ShareGPT 类通用聊天上，EAGLE-3（在 ShareGPT 上训练）达到 0.6-0.8。在领域特定流量（代码、医疗、法律）上，通用数据训练的草稿头跌到 0.4-0.6。训练领域特定的 EAGLE-3 草稿头可以恢复 α——这是比目标模型微调轻得多的训练任务。

### 2.3 EAGLE 代际概览

- **经典 draft model：** 同系列小模型。α 0.3-0.5。基础设施简单——两个模型加载，草稿每次做 K 次前向。
- **EAGLE-1（2024）：** 在目标隐藏状态（最后一层）上训练的轻量草稿头。α ~0.5-0.6。目标模型上额外的参数开销很小。
- **EAGLE-2（2025）：** 自适应草稿长度+树状草稿（一次目标前向验证多分支）。α ~0.6-0.7。更复杂的草稿调度器。
- **EAGLE-3（2025-2026）：** 在目标模型的多个层（不只是最后一层）上训练草稿头。通用聊天上 α ~0.6-0.8。

### 2.4 2026 年生产配方

1. 先上线基础模型。测量基线 TTFT、ITL、吞吐量
2. 通过 vLLM `speculative_config` 启用 EAGLE-3 草稿。重跑基准测试
3. 记录接受率 α。vLLM V1 将其报告为 `spec_decode_metrics.accepted_tokens_per_request`——除以请求的草稿长度即得 α
4. 如果生产流量分布上 α < 0.55，禁用投机解码或训练领域特定的 EAGLE-3 草稿
5. 在生产并发下重跑。确认 P99 ITL 没有变差

### 2.5 生产陷阱：P99 尾部

投机解码降低了平均 ITL，但 P99 可能变差——如果被拒绝的草稿触发了两遍序列（draft + verify-fail + 重滚）。在满负载下，这两遍是串行的。**看 P99 ITL，不要只看 P50。**

### 2.6 什么时候不用投机解码

- Batch-1 离线生成，延迟不重要——用纯目标
- 极短输出（少于 50 词元）——草稿开销和验证成本主导
- 没有领域训练草稿头的专用领域——α 太低
- vLLM v0.18.0 + draft-model 投机 + Chunked Prefill——不编译

### 2.7 盈亏平衡公式

预期加速比：`S(α, K) = (1 + K×α) / (1 + verify_overhead)`。令 S=1 解出 α：`α_breakeven = verify_overhead / K`。典型 verify_overhead ~0.15、K=5 时：α_breakeven = 0.03。但这是原始 decode 数学。在高并发下 verify 开销上升且 decode 批次已经分摊了内存读取，有效的 α_breakeven 实际达到 ~0.45-0.55。

## 3. 从零实现

### 第 1 步：投机解码加速比模拟

```python
def spec_speedup(alpha: float, K: int, verify_overhead: float = 0.15):
    """计算投机解码的预期加速比。"""
    expected_tokens = 1 + K * alpha
    total_cost = 1 + verify_overhead
    return expected_tokens / total_cost


def find_breakeven_alpha(K: int, verify_overhead: float = 0.15):
    """找出盈亏平衡的 α。"""
    return verify_overhead / K


# 不同 (alpha, K) 组合的加速比
for alpha in [0.3, 0.4, 0.5, 0.6, 0.7, 0.8]:
    for K in [3, 5, 8]:
        s = spec_speedup(alpha, K)
        be = find_breakeven_alpha(K)
        print(f"α={alpha:.1f} K={K} → 加速比={s:.2f}x  盈亏平衡α={be:.3f}")
```

完整代码见 `code/main.py`。

## 5. 工程最佳实践

### 5.1 先测量 α，再翻开关

永远不要在不测量的生产流量上开启投机解码。先上基础模型，测量基线。然后启用，重测。α < 0.55 就禁用。

### 5.2 领域特定草稿头

如果你的流量 30% 是代码、30% 是医疗、40% 是通用聊天——在通用聊天上 α 可能是 0.7，但代码和医疗上可能只有 0.4。混合后的 α 是 0.58——接近盈亏平衡。训练领域特定的草稿头可以将代码的 α 提升到 0.6+。

### 5.3 P99 是你该看的指标

投机解码降低了 P50 ITL，但 P99 可能因为拒绝重滚而上升。只看 P50 是一个陷阱。

### 5.4 中文场景特别建议

- **中文草稿头训练。** EAGLE-3 的草稿头在中文流量上可能有不同的 α。中文的词元密度更高——同样的语义信息需要更少的词元。草稿头在中文上可能更有效（因为每次验证的词元更少）
- **EAGLE-3 草稿头训练成本。** 在中文数据上训练 EAGLE-3 草稿头需要 1-2 天的 GPU 时间（A100 级别），远低于目标模型微调

## 6. 常见错误

### 错误 1：不测量就开启投机解码

**现象：** 开启 EAGLE-3 后，平均 ITL 下降 20%，但 P99 ITL 上升 30%。用户体验反而变差。

**原因：** 流量中 40% 是代码生成，α 只有 0.4。被拒绝的草稿触发了大量两遍序列，P99 尾部恶化。

**修复：** 先测量 α。α < 0.55 就不要开启。或训练领域特定草稿头。

### 错误 2：在短输出上启用投机解码

**现象：** 投机解码开启后，简单分类任务（输出 5-10 词元）的延迟反而增加了。

**原因：** 对于极短输出，草稿开销（加载 draft head + 草稿前向）超过了节省的 decode 步数。

**修复：** 为不同输出长度配置不同策略——短输出禁用投机，长输出启用。

## 7. 面试考点

### Q1：为什么 α 低于 0.55 时投机解码是净负面的？（难度：⭐⭐⭐）

**参考答案：**
被拒绝的草稿触发两遍序列——draft + verify-fail + 重滚。每拒绝一个词元，目标模型多做一次前向传播。在高并发下，decode 批次已经足够大——多个序列共享同一批权重读取——"目标单独 decode"和"目标+验证"之间的内存带宽差距变小。当 α < 0.55 时，拒绝重滚的额外前向传播成本超过了接受词元的节省。净效果是：更多的计算做同样的工作。

### Q2：EAGLE-3 相比经典 draft model 为什么 α 更高？（难度：⭐⭐）

**参考答案：**
经典 draft model 使用同系列的小模型（如 Llama 3.2 1B）——小模型的分布与大模型分布差异显著，导致很多草稿词元被拒绝。EAGLE-3 在目标模型的隐藏状态上训练轻量草稿头——它"看到"了目标模型在相同输入上的内部表示，因此草稿词元的分布更接近目标模型的分布，α 从 0.3-0.5 提升到 0.6-0.8。

## 🔑 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 投机解码 | "草稿+验证" | 草稿提出 K 个词元，目标在一次前向中验证全部 |
| 接受率 α | "投机接受率" | 被目标接受的草稿词元比例；唯一重要的指标 |
| 草稿长度 K | "投机 K" | 草稿每次提出的词元数；典型值 4-8 |
| EAGLE-3 | "最新 EAGLE" | 在目标模型多个层上训练草稿头；通用聊天 α 0.6-0.8 |
| `speculative_config` | "vLLM 投机配置" | vLLM V1 中的显式 opt-in；不设就没有加速 |
| 盈亏平衡 α | "无用的 α" | 投机解码给出零加速时的 α；在生产并发下监控 |
| 拒绝草稿两遍 | "重滚成本" | 拒绝时目标前向执行两次；驱动 P99 尾部 |

## 📚 小结

投机解码是用廉价草稿换取廉价 decode 的交易——关键指标是接受率 α。EAGLE-3 通过在目标隐藏状态上训练草稿头将 α 提升到 0.6-0.8。但 α 因流量分布而异——领域特定流量可能只有 0.4。α 低于约 0.55 时投机解码是净负面。vLLM 2026 默认关闭投机解码——先测量，再翻开关。关注 P99 ITL，不只是 P50。
