## 1. 问题

值班工程师在凌晨 3 点收到页面。"结账错误率飙升。"他们查看 Datadog、Loki、三个手册、部署日志。30 分钟后才意识到根因是 vLLM OOM（KV 缓存尖峰）。重启 pod 后错误清除。

在 2026 年，这 20 分钟的调查是可自动化的。按服务分组日志、关联到最近部署、匹配手册——都是 RAG + 工具使用。监督智能体可以在人类打开 Datadog 之前做第一轮分类并提出假设。

完全自主修复是不同的问题。重启 pod：安全。扩展 GPU 池：在策略允许下安全。重新架构服务：绝对不行。纪律是划定那条窄线。

## 2. 概念

### 2.1 多智能体架构

```
          事故
             │
             ▼
        监督者
        /    |    \
       ▼     ▼     ▼
  日志智能体 指标智能体 手册智能体
       │     │     │
       └─────┴─────┘
             │
             ▼
        假设 + 证据
             │
             ▼
        人类审批
             │
             ▼
        动作（窄范围）
```

监督者将事故拆分为子查询。专业化智能体有工具访问权（日志搜索、PromQL、文档检索）。监督者综合，向人类呈现假设+证据。人类批准或重新定向。

### 2.2 自主修复范围

**安全（窄范围）：** 重启 pod、回滚特定部署、在预批准范围内扩展池、启用预批准的 feature flag。

**不安全（广范围）：** 改变服务拓扑、修改资源限制、部署新代码、修改 IAM、更改数据库。

任何卖"设置后就不管"的人都是在过度销售。安全集随着 AI SRE 成熟而增长，但边界是真实的。

### 2.3 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事故。如果它们对根因一致，置信度高。如果不一致，将两个假设都展示给人类。简单模式，有效过滤幻觉根因。

### 2.4 运营记忆

团队换班是传统 SRE 的沉默杀手——部落知识离开。AI SRE 将手册+事后分析存储在向量数据库中；智能体在每次新事故时检索。新工程师加入时，AI 拥有完整历史。

### 2.5 事件前预测

MIT 2025 研究：在历史日志、GPU 温度、API 错误模式上训练的 LLM，在测试集上提前 10-15 分钟预测了 89% 的宕机。

现实检查：没有执行的预测只是仪表盘。运营问题是"当我们预测时，做什么？"预排空？通知？自动扩缩？答案取决于策略。

## 3. 从零实现

### 第 1 步：多智能体事故分类模拟

```python
def simulate_triage(log_errors, metric_anomaly, runbook_match):
    """模拟多智能体事故分类。"""
    hypotheses = []

    if log_errors:
        hypotheses.append({
            "source": "日志智能体",
            "hypothesis": f"检测到 {len(log_errors)} 个错误模式",
            "confidence": 0.8,
        })

    if metric_anomaly:
        hypotheses.append({
            "source": "指标智能体",
            "hypothesis": "检测到 CPU/内存异常",
            "confidence": 0.75,
        })

    if runbook_match:
        hypotheses.append({
            "source": "手册智能体",
            "hypothesis": f"匹配已知模式: {runbook_match}",
            "confidence": 0.9,
        })

    # 监督者评估：是否所有智能体一致？
    if len(hypotheses) >= 2:
        opinions = [h["hypothesis"] for h in hypotheses]
        consistent = len(set(opinions)) == 1
    else:
        consistent = False

    return {
        "hypotheses": hypotheses,
        "consistent": consistent,
        "action": "执行修复" if consistent else "升级给人类",
    }


# 演示
result = simulate_triage(
    log_errors=["vLLM OOM", "KV cache spike"],
    metric_anomaly=True,
    runbook_match="vLLM OOM - 扩容 GPU",
)
print(f"假设数量: {len(result['hypotheses'])}")
print(f"一致: {result['consistent']}")
print(f"动作: {result['action']}")
for h in result["hypotheses"]:
    print(f"  {h['source']}: {h['hypothesis']} (置信度: {h['confidence']})")
```

完整代码见 `code/main.py`。

## 5. 工程最佳实践

### 5.1 AI 建议，人类批准

完全自主修复是窄范围的——重启 pod、回滚部署、在预批准范围内扩展。任何更复杂的事情都需要人类批准。

### 5.2 对抗性评估是低成本高质量

两个模型独立分析同一事故，一致=高置信度，不一致=升级给人类。实现成本低但过滤幻觉效果好。

### 5.3 中文场景特别建议

- **国内 AI SRE 产品的可用性。** Datadog 和 Azure 在国内的 AI SRE 功能可能受限。国内团队可能需要自建或使用阿里云 ARMS、华为云 AOM 等国内 APM
- **中文手册的 RAG 检索。** 中文手册需要中文向量嵌入模型。bge-large-zh 是常用选择
- **事件前预测的中文数据。** 国内 LLM 的日志格式和错误模式与英文不同——事件前预测模型需要在中文日志上训练

## 6. 常见错误

### 错误 1：AI 建议后直接执行

**现象：** AI 建议重启 pod，系统自动执行。但重启的是一个正在处理支付的智能体——导致支付中断。

**原因：** AI 建议没有经过人类审批。"设置后就不管"在 LLM 时代是危险的。

**修复：** AI 建议→人类审批→执行。只有极窄范围的操作（重启空闲 pod）可以免审批。

### 错误 2：没有运营记忆

**现象：** 资深工程师离职后，新团队遇到同样的宕机无法解决。

**原因：** 部落知识没有被记录和索引。

**修复：** 每次事后分析后将手册+教训存入向量数据库。新事故时智能体自动检索历史。

## 7. 面试考点

### Q1：AI SRE 的多智能体架构是什么？（难度：⭐⭐）

**参考答案：**
监督者智能体作为顶层协调——接收事故信息，拆分为子查询，分发给专业化智能体（日志、指标、手册）。每个专业化智能体有工具访问权（日志搜索、PromQL、文档检索）。监督者综合所有假设和证据，向人类呈现假设+证据。人类批准或重新定向。然后执行窄范围的修复（重启 pod、回滚部署等）。

### Q2：MIT 的 89% 提前检测结果的实际约束是什么？（难度：⭐⭐⭐）

**参考答案：**
预测没有执行只是仪表盘。运营问题是"当我们预测到 10-15 分钟后可能宕机时，做什么？"三个选项：预排空（在问题发生前迁移流量）、通知（pager 通知值班人员）、自动扩缩（提前扩展 GPU 池）。每个选项都有成本和副作用——预排空可能影响正在处理的请求，自动扩缩要付费。答案取决于策略——没有一刀切的解决方案。

## 📚 小结

AI SRE 使用多智能体架构——监督者协调专业化智能体（日志、指标、手册），向人类呈现假设+证据。自主修复保持窄范围（重启 pod、回滚部署），广范围操作需要人类批准。对抗性评估（两个模型一致=高置信度）是低成本高质量的过滤器。事件前预测（MIT 89% 提前检测）是前沿方向，但没有执行的预测只是仪表盘。
