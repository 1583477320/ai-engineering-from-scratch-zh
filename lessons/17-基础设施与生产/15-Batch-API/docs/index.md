# Batch API——50% 折扣作为行业标准

> 每个主要提供商都发布异步批处理 API，50% 折扣，约 24 小时交付。OpenAI、Anthropic、Google 和大多数推理平台（Fireworks 批处理层、Together 批处理）实现相同模式。将批处理与提示词缓存叠加，夜间流水线可以降到同步无缓存成本的约 10%。规则残酷地简单：如果不是交互式的，就应该上批处理。内容生成流水线、文档分类、数据提取、报告生成、批量标注、目录打标签——任何能容忍 24 小时延迟的工作负载在移到批处理之前都是在桌上留钱。

## 1. 问题

你的团队发布了夜间报告生成流水线。50,000 个文档，逐个摘要，聚类摘要，起草执行简报。同步运行需要 4 小时，每晚 $2,000。你听说了批处理 API。

批处理给你 50% 折扣。你还在系统提示词上启用了提示词缓存（跨所有 50K 次调用共享）。叠加后，账单降到每晚 $180——基线的约 9%。同样的流水线，三个配置变更。

批处理是 LLM 成本工具箱中没有人拉的最便宜的杠杆。原因主要是组织性的：团队在 SLA 实际是"明早之前"时却想着"实时"。

## 2. 概念

### 2.1 三大批处理 API

**OpenAI Batch API：** JSONL 文件上传，请求列表。承诺 24 小时交付（实际通常 2-8 小时）。输入和输出词元 50% 折扣。`/v1/batches` 端点。缓存符合条件的输入还能叠加缓存输入定价。

**Anthropic Message Batches：** JSONL 上传。24 小时交付。50% 折扣。支持 `cache_control`——缓存写入是显式的，批处理内自动读取。

**Google Vertex AI Batch Prediction：** BigQuery 或 GCS 输入。Gemini 同样的 50% 折扣。与 Vertex 流水线集成。

### 2.2 语义：异步，不是慢

批处理是"我承诺 24 小时内返回"——不是"这需要 24 小时"。典型 P50 是 2-6 小时。提供商在 GPU 利用率低的非高峰期调度你的批处理。

### 2.3 与缓存叠加

50K 文档摘要 + 相同的 4K 词元系统提示词：

- 同步无缓存：50000 ×（$input × 4000 + $output × 200）全价
- 同步有缓存：系统提示词第一次写入后缓存；剩余 49999 次输入便宜 10 倍
- 批处理+缓存：以上全部 + 读写均 50% 折扣

叠加效果：批处理+缓存 = 同步无缓存账单的约 10%。

### 2.4 工作负载分流

**交互式——** 用户等待响应。TTFT 重要。同步调用+提示词缓存。不能批处理。

**半交互式——** 用户提交任务，几分钟后查看。异步队列+批处理不可用时回退到同步。中等规模的 RAG 索引。

**批处理——** 用户期望"明早"或"下个小时"的结果。内容流水线、大规模分类、离线分析。永远批处理，永远叠加缓存。

常见错误：因为流水线是生产环境就将所有东西分类为交互式。**生产不是延迟规范——SLA 才是。**

### 2.5 部分交互性陷阱

有些功能看起来是交互式的但能容忍 5-10 分钟。例子：带"刷新"按钮的夜间客户健康报告。用户点击刷新；等 10 分钟没问题。团队将其作为同步实现。50 个并发刷新的成本是批处理+邮件交付的 10 倍。

要问的问题："24 小时对这个用户意味着什么？"如果答案是"他们不会注意到"，就批处理。

### 2.6 输出格式陷阱

批处理文件格式每提供商不同：

- OpenAI：JSONL，每行一个请求
- Anthropic：JSONL，每行一个消息；响应格式内嵌
- Vertex：BigQuery 表或 TFRecord 的 GCS 前缀

跨提供商写"一个批处理客户端"意味着每个提供商的适配代码。

## 3. 从零实现

### 第 1 步：批处理 vs 同步成本对比

```python
def compare_batch_sync(documents, input_tokens, output_tokens,
                       input_rate, output_rate, batch_discount=0.5):
    """对比同步和批处理的成本。"""
    # 同步无缓存
    sync_cost = documents * (input_tokens * input_rate + output_tokens * output_rate) / 1_000_000

    # 同步有缓存（第一次写入后共享前缀 10x 便宜）
    first_write = 1 * input_tokens * input_rate / 1_000_000
    cached_reads = (documents - 1) * input_tokens * input_rate * 0.1 / 1_000_000
    sync_cached = first_write + cached_reads + documents * output_tokens * output_rate / 1_000_000

    # 批处理 + 缓存
    batch_cached = sync_cached * (1 - batch_discount)

    return {
        "sync": sync_cost,
        "sync_cached": sync_cached,
        "batch_cached": batch_cached,
        "savings_pct": 1 - batch_cached / sync_cost,
    }


r = compare_batch_sync(50000, 4000, 200, 3.0/1_000_000, 15.0/1_000_000)
print(f"同步无缓存: ${r['sync']:.0f}")
print(f"同步有缓存: ${r['sync_cached']:.0f}")
print(f"批处理+缓存: ${r['batch_cached']:.0f}")
print(f"节省: {r['savings_pct']:.1%}")
```

完整代码见 `code/main.py`。

## 5. 工程最佳实践

### 5.1 永远先分流

每个新的 LLM 工作负载先问：这个 SLA 需要同步吗？如果 24 小时可接受，就批处理。

### 5.2 叠加批处理 + 缓存

这是成本最低的组合。共享系统提示词的夜间工作负载可以降到基线的约 10%。

### 5.3 中文场景特别建议

- **国内批处理 API。** 阿里云百炼、百度千帆等国内平台的批处理支持参差不齐。使用前确认批处理 API 是否可用
- **批处理延迟。** 国内云厂商的批处理交付时间可能比 OpenAI/Anthropic 长（12-48 小时 vs 2-8 小时）
- **中文提示词缓存叠加。** 中文批处理+缓存的效果可能比英文更好——中文系统提示词通常更长，缓存收益更大

## 6. 常见错误

### 错误 1：所有工作负载都分类为交互式

**现象：** 夜间报告生成用同步运行，每晚 $2,000。

**原因：** 团队认为"生产环境"必须是实时的。但用户只在早上查看报告。

**修复：** 问"24 小时对用户意味着什么？"如果答案是"他们不会注意到"，就批处理。

### 错误 2：不叠加缓存

**现象：** 用了批处理但账单只降了 50%。

**原因：** 没有启用提示词缓存。50K 文档中每个都重新 prefill 系统提示词。

**修复：** 批处理+缓存叠加：50% 折扣 × 10 倍缓存折扣 = 成本降到基线的约 10%。

## 7. 面试考点

### Q1：批处理+缓存叠加为什么能降到基线的 10%？（难度：⭐⭐⭐）

**参考答案：**
两层折扣叠加。L2 缓存：共享系统提示词第一次写入后，剩余 49999 次读取便宜 10 倍——输入成本从 100% 降到约 10%。批处理折扣：所有输入和输出 50% 折扣。叠加效果：输入成本 = 10% × 50% = 5%，输出成本 = 50%。对于长提示词短输出的工作负载（如分类），总成本约 10%。对于短提示词长输出的工作负载，节省更少（输出占主导）。

### Q2：如何判断一个工作负载应该批处理还是同步？（难度：⭐⭐）

**参考答案：**
核心判断标准是 SLA，不是"生产环境"标签。问用户："24 小时内拿到结果可以接受吗？"如果答案是"可以"（如夜间报告、批量分类、离线分析），就批处理。如果用户在等待结果（聊天、代码补全、语音交互），就同步。很多团队错误地将所有生产工作负载分类为交互式——结果是在不需要实时的地方为实时付费。

## 🔑 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| Batch API | "异步折扣" | 50% 折扣 + 24 小时交付 |
| JSONL | "批处理格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |

| 工作负载分流 | "交互性决策" | 交互式/半交互式/批处理路由决策 |
| 叠加折扣 | "批处理+缓存" | 两者同时应用时约 10% 的无缓存同步成本 |

## 📚 小结

批处理 API 是 LLM 成本工具箱中最便宜的杠杆——50% 折扣+24 小时交付。叠加提示词缓存后，夜间工作负载可以降到基线的约 10%。关键判断标准是 SLA——"24 小时可接受吗？"如果是，就批处理。将每个新工作负载分流为交互式/半交互式/批处理。不要因为"这是生产环境"就假定需要实时——生产不是延迟规范，SLA 才是。
