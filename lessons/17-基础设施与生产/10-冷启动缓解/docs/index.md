# Serverless LLM 冷启动缓解

> 一个 20GB 的模型镜像从冷到热需要 3-8 分钟（7B）到 20+ 分钟（70B）。在真正的 serverless 世界里，那不是热身——那是宕机。缓解方案在五个层面上运作：预播种节点镜像、模型流式加载、GPU 内存快照、热池、分层加载。Modal 的 2-4 秒冷启动是一个可以追求的基准线。

**预计时间：** ~60 分钟
**所处阶段：** Tier 3

## 1. 问题

你的 serverless LLM 端点在夜间缩到零。早上 8 点流量激增。第一个请求在等待：

1. Karpenter 拉 GPU 节点：45-60 秒
2. 容器拉取 30GB 镜像（含权重）：120-300 秒
3. 引擎将权重加载到 HBM：45-120 秒（取决于模型大小和存储速度）
4. vLLM 或 TRT-LLM 初始化 CUDA 图、KV 缓存池、分词器：10-30 秒

总计：220-510 秒（约 3-8 分钟）才能返回第一个词元。你的 SLA 是 2 秒。你部署了热池（`min_workers=1`），问题看似消失了——但现在你要为一台空闲的 GPU 支付 24x7 的费用。

冷启动缓解就是在保持 serverless 经济性的同时，逼近始终在线的延迟。

## 2. 概念

### 2.1 第 1 层——预播种节点镜像（Bottlerocket）

在 AWS 上，Bottlerocket 的双卷架构将操作系统与数据分开。用预拉取的容器镜像快照数据卷；在 `EC2NodeClass` 中引用快照 ID。新节点启动时权重已在本地 NVMe 上。典型节省：大型模型每次冷启动节省 2-4 分钟。GCP 等效：预烘焙容器层的自定义 VM 镜像。Azure 等效：托管磁盘快照。

### 2.2 第 2 层——模型流式加载（Run:ai Model Streamer）

不用在回答第一个请求之前加载完整文件，而是逐层将权重流式传输到 GPU 内存——一旦第一个 transformer 块加载完成就开始处理。NVIDIA Run:ai Model Streamer 原生集成在 vLLM 2026 中。支持 S3、GCS 和本地 NVMe。通过将 I/O 与计算设置重叠，大体上将大型模型的权重加载时间减半。

### 2.3 第 3 层——GPU 内存快照（Modal）

Modal 在第一次加载后对 GPU 状态（权重、CUDA 图、KV 缓存区域）做检查点。后续重启直接反序列化到 HBM——比重新初始化快 10 倍。这是最接近"在 2 秒内启动一个热 GPU"的方案。权衡：快照是每 GPU 拓扑的——如果 Karpenter 将你迁移到不同 SKU，你需要重新检查点。

### 2.4 第 4 层——热池（min_workers=1）

最简单的缓解方案：保持一个副本始终就绪。成本是一台 GPU 的 24x7 小时费率。小模型上对比很尖锐（每小时 $0.85-$1.50 避免 30 秒冷启动），大模型上则合理（每小时 $4 避免 5 分钟冷启动）。热池成为强制性的 SLA 阈值：通常在 70B+ 模型上 TTFT P99 < 60 秒。

### 2.5 第 5 层——分层加载（ServerlessLLM）

ServerlessLLM 将存储视为层级结构：NVMe（快但大）、DRAM（中等但分层）、HBM（极小但即时）。权重预加载到 DRAM；按需加载到 HBM。论文报告冷加载延迟比朴素的硬盘到 HBM 降低 10-200 倍。生产采用尚早但已有与 vLLM 的集成。

### 2.6 第 6 层——实时迁移（额外模式）

当节点不可用时（Spot 驱逐、节点排水），传统模式是冷启动另一个副本并排空请求队列。实时迁移将输入词元（KB）传输到已加载模型的目标，并在目标上重计算 KV 缓存。重计算比通过网络传输 GB 级的 KV 缓存更便宜。适用于分离式部署。

### 2.7 热池的数学

对于一个 P99 TTFT SLA 为 2 秒的服务，问题不是"要不要热池"而是"要几个热副本、哪些路径需要"。

- 高价值交互路径（实时聊天、语音智能体）：`min_workers=1-2`
- 后台批处理路径（夜间分类）：接受缩到零，5-10 分钟冷启动可容忍
- 高级套餐：每租户 `min_workers` 加专用容量

### 2.8 先测量再优化

70B 模型在全新节点上的冷启动解剖：

| 阶段 | 时间 | 缓解方案 |
|---|---|---|
| 节点供应 | 50s | Bottlerocket + 预播种镜像、热池 |
| 镜像拉取 | 180s | 预播种数据卷（消除） |
| 权重到 HBM | 75s | 模型流式加载（减半）；GPU 快照（消除） |
| 引擎初始化 | 20s | 持久化 CUDA 图缓存 |
| 首次前向 | 3s | 最小固有延迟 |
| **冷启动总计** | **328s** | |
| **缓解后总计** | **~15s** | **22 倍减少** |

## 3. 从零实现

### 第 1 步：冷启动时间分解

```python
def cold_start_anatomy(model_size: str, with_mitigations=False):
    """计算冷启动的各阶段时间。"""
    anatomy = {
        "7B": {"node": 50, "image": 120, "weight_load": 30, "engine": 15, "forward": 2},
        "13B": {"node": 50, "image": 150, "weight_load": 45, "engine": 18, "forward": 2.5},
        "70B": {"node": 50, "image": 180, "weight_load": 75, "engine": 20, "forward": 3},
    }

    a = anatomy.get(model_size, anatomy["7B"])
    total = sum(a.values()) / 60  # 转为分钟

    if with_mitigations:
        # Bottlerocket 消除镜像拉取 + 模型流式加载减半权重加载
        a["image"] = 0
        a["weight_load"] = a["weight_load"] * 0.5
        total_mitigated = sum(a.values()) / 60
        return total, total_mitigated

    return total, None


for size in ["7B", "13B", "70B"]:
    raw, mitigated = cold_start_anatomy(size)
    if mitigated:
        print(f"{size:4s}: 原始 {raw:.1f} 分钟 → 缓解后 {mitigated:.1f} 分钟")
    else:
        print(f"{size:4s}: {raw:.1f} 分钟")
```

完整代码见 `code/main.py`。

## 4. 工业工具

### 4.1 缓解方案对照

| 层 | 工具/模式 | 节省 | 适用场景 |
|---|---|---|---|
| 节点镜像 | Bottlerocket 预播种 | 消除镜像拉取（2-3 分钟） | AWS 上的大型模型 |

| GPU 快照 | Modal 检查点 | 重启快 10 倍 | Serverless 平台 |
| 热池 | min_workers=1 | 消除冷启动（有空闲成本） | SLA 关键路径 |
| 分层加载 | ServerlessLLM | 10-200 倍延迟降低 | NVMe+DRAM+HBM 场景 |

## 5. 工程最佳实践

### 5.1 按路径分层热池

交互路径（实时聊天、语音）：`min_workers=1-2`。批处理路径（离线分类）：缩到零。高级租户：每租户专用热副本。不要对所有路径一视同仁。

### 5.2 组合使用而非单选

没有单一方案能解决所有冷启动问题。Bottlerocket 消除镜像拉取 + Streamer 加速权重加载 + 热池消除最坏情况尾部 = 22 倍减少。

### 5.3 中文场景特别建议

- **国内节点供应时间更长。** 阿里云、华为云的 GPU 节点的供应时间通常比 AWS 长（60-90 秒 vs 45-60 秒）。热池对于国内部署更重要
- **国内镜像拉取速度受限。** 国内从海外 registry 拉取模型镜像的速度可能很慢（5-10 分钟）。使用阿里云 ACR、华为云 SWR 等国内镜像仓库，或预播种节点镜像
- **Modal 类型服务国内缺失。** 国内缺乏 `min_workers` + GPU 快照类型的 serverless LLM 平台。大部分场景需要自建热池

## 6. 常见错误

### 错误 1：不测量就优化

**现象：** 部署了 GPU 快照，但冷启动时间没有明显改善。检查发现瓶颈在节点供应（50 秒）而不是权重加载——快照没有帮助。

**原因：** 没有先测量冷启动的各阶段时间分布。

**修复：** 先测量。如果瓶颈在节点供应，预播种镜像或热池才是正确的方案。

### 错误 2：所有路径都用 min_workers=1

**现象：** 5 个服务的路径都设了 `min_workers=1`，每月 GPU 小时数暴增。

**原因：** 非关键路径不需要热池。

**修复：** 按路径分层——交互路径用热池，批处理路径缩到零。

## 7. 面试考点

### Q1：冷启动的五层缓解方案是什么？哪一层效果最大？（难度：⭐⭐）

**参考答案：**
五层：节点镜像（Bottlerocket 预播种）、模型流式加载（Run:ai Model Streamer）、GPU 快照（Modal 检查点）、热池（min_workers=1）、分层加载（ServerlessLLM）。效果最大的方案取决于瓶颈在哪——如果瓶颈是节点供应（50 秒） + 镜像拉取（180 秒） + 权重加载（75 秒），Bottlerocket（消除镜像拉取） + 热池（消除节点供应）的组合效果最大。但最直接的方法是热池——只要愿意付空闲 GPU 的钱。

### Q2：实时迁移为什么传输输入词元而不是 KV 缓存？（难度：⭐⭐⭐）

**参考答案：**
KV 缓存是 GB 级的（128K 上下文 × 80 层 × 8 头 = 数百 MB 到数 GB），通过网络传输延迟高且带宽浪费。输入词元是 KB 级的（通常 < 0.01 GB），传输极快——但代价是接收端需要从头重计算 KV 缓存。这个重计算约 800ms（对于长提示）——比传输数 GB KV 缓存快得多。权衡：传输 KB + 重计算 800ms vs 传输 GB（可能需要数秒）。在小提示上重计算更划算。

## 🔑 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 冷启动 | "大停顿" | 全新副本上从请求到首个词元的时间 |
| 热池 | "始终在线" | min_workers ≥ 1 保持至少一个副本就绪 |
| 预播种镜像 | "烘焙 AMI" | 权重预置的节点镜像 |
| Bottlerocket | "AWS 节点 OS" | AWS 容器优化 OS，支持双卷快照 |
| 模型流式加载 | "流式加载" | I/O 与计算设置重叠 |
| GPU 快照 | "检查点到 HBM" | 序列化后加载的 GPU 状态；重启时反序列化 |
| 分层加载 | "NVMe+DRAM+HBM" | 存储层级；按需加载 |

## 📚 小结

冷启动缓解在五个层面上运作：节点镜像（消除镜像拉取）、模型流式加载（重叠 I/O 与计算）、GPU 快照（10 倍重启速度）、热池（消除最坏情况尾部）、分层加载（10-200 倍延迟降低）。没有单一方案是银弹——组合使用效果最好。关键是先测量瓶颈在哪，再选择对应层的方案。热池是最直接但也最贵的方案——按路径分层使用。

下一课将讨论多区域 LLM 服务与 KV 缓存局部性——当你的服务扩展到多个区域时，路由策略决定了用户体验。
