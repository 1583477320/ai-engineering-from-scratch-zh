# 提示词缓存与语义缓存经济学

> 缓存在两个层面发生。L2（提供商级）提示词/前缀缓存复用重复前缀的注意力 KV——Anthropic 的提示词缓存文档宣称在长提示上成本降低 90%、延迟降低 85%。L1（应用级）语义缓存在嵌入相似度命中时完全跳过 LLM。两个陷阱：并行化杀死缓存（N 个并行请求在第一个缓存写入之前发出，账单膨胀 5-10 倍），动态内容在前缀内阻止缓存命中。ProjectDiscovery 报告通过将动态文本移出可缓存前缀，命中率从 7% 提升到 74%。

## 1. 问题

你为 RAG 服务添加了提示词缓存。账单持平。你测量命中率——只有 7%。你的提示词看起来是静态的但其实不是——系统提示词包含当前精确到分钟的日期、一个请求 ID、以及一个随机化的示例重排。每个请求都写入一个新的缓存条目，读取零。

另外，你的智能体每个用户问题运行 10 个并行工具调用。10 个都到达提供商时，第一个缓存写入还没完成。10 次写入，0 次读取。你的账单是"有缓存"版本的 5-10 倍。

**缓存是一个协议，不是一个开关。两层，两种不同的失败模式。**

---

## 2. 概念

### 2.1 L2——提供商提示词/前缀缓存

提供商存储可缓存前缀的注意力 KV，在下一个匹配前缀的请求上复用。你支付一次写入成本，读取几乎免费。

**Anthropic（Claude 3.5/3.7/4 系列）：** 显式 `cache_control` 标记。TTL：5 分钟（写入成本 1.25 倍基础价）或 1 小时（写入成本 2 倍）。缓存读取：Claude 3.5 Sonnet 上 $0.30/M vs $3.00/M 新鲜——10 倍便宜。

**OpenAI：** 对 ≥1024 词元的提示词自动缓存。缓存输入约为新鲜输入的 10%。社区报告的命中率约 30-60%。

**Google（Gemini）：** 通过显式 API 的上下文缓存；1M 词元上下文使缓存收益更大。

**自托管（vLLM、SGLang）：** 第 17 · 06 课覆盖了 RadixAttention——同一模式在你自己的计算上。

### 2.2 L1——应用级语义缓存

在调用 LLM 之前，哈希提示词、嵌入它、查找相似的缓存请求（余弦相似度 > 阈值，通常 0.95+）。命中时返回缓存响应。未命中时调用 LLM 并缓存结果。

开源方案：Redis Vector Similarity、GPTCache、Qdrant。商业方案：Portkey Cache、Helicone Cache。

**生产命中率：**
- 开放式聊天：10-15%
- 结构化 FAQ/支持：40-70%
- 代码问题：20-30%（小变体会杀死命中）
- 语音智能体重复提示词：50-80%（语音归一化固定集合）

### 2.3 并行化反模式

你的智能体对每个用户问题做 10 个并行工具调用。10 个都有相同的 4K 词元系统提示词。Anthropic 的缓存写入是逐请求的；第一次缓存写入在提供商看到提示词后约 300ms 完成。请求 2-10 在同一毫秒窗口到达，每次都看到缓存未命中。你支付了 10 次写入溢价，0 次读取折扣。

**修复：** 先顺序发出请求 1，等缓存填充后再并行发出 2-10。给第一个工具调用增加 300ms；节省 5-10 倍账单。

### 2.4 动态内容反模式

你的系统提示词：

```
You are a helpful assistant. The current time is 14:32:17.
User ID: abc123. Today is Tuesday...
```

每个请求都是唯一的。每个请求都写入。零命中。

**修复：** 将真正静态的部分移到可缓存前缀；将动态内容追加到缓存边界之后：

```
[cacheable]
You are a helpful assistant. [rules, examples, instructions]
[/cacheable]
[dynamic, not cached]
Current time: 14:32:17. User: abc123.
```

ProjectDiscovery 通过这种方式将命中率从 7% 提升到 74%。

## 3. 从零实现

### 第 1 步：双层缓存模拟器

```python
def l2_cache_cost(requests, prompt_tokens, cached_pct=0.5,
                  fresh_rate=3.0, cached_rate=0.30, write_premium=1.25):
    """L2 提示词缓存的成本计算。"""
    fresh_requests = int(requests * (1 - cached_pct))
    cached_requests = int(requests * cached_pct)
    write_cost = cached_requests * prompt_tokens * fresh_rate * write_premium / 1_000_000
    read_cost = cached_requests * prompt_tokens * cached_rate / 1_000_000
    no_cache_cost = requests * prompt_tokens * fresh_rate / 1_000_000
    total = fresh_requests * prompt_tokens * fresh_rate / 1_000_000 + write_cost + read_cost
    return {
        "no_cache": no_cache_cost,
        "with_cache": total,
        "savings": 1 - total / no_cache_cost,
    }


# 演示
r = l2_cache_cost(100_000, 4000, cached_pct=0.6)
print(f"无缓存: ${r['no_cache']:.2f}")
print(f"有缓存: ${r['with_cache']:.2f}")
print(f"节省: {r['savings']:.1%}")
```

完整代码见 `code/main.py`。

## 5. 工程最佳实践

### 5.1 将动态内容移出可缓存前缀

这是最简单、最有效的缓存优化。ProjectDiscovery 的案例：7% → 74% 命中率。

### 5.2 避免并行化反模式

并行工具调用时，先顺序发出第一个请求让缓存填充，再并行发出其余。给第一个请求增加 300ms；节省 5-10 倍。

### 5.3 中文场景特别建议

- **Anthropic 和 OpenAI 的缓存对中文有效。** 中文提示词的前缀复用率可能更高——中文系统提示词通常更长且更固定
- **语义缓存在中文上更有效。** 中文的同义表达多——"机器学习"和"ML"是同一概念。语义缓存可以捕获这些变体
- **国内 LLM 的缓存支持。** 阿里云百炼、百度千帆等国内平台的缓存支持参差不齐。使用前确认缓存是否可用

## 6. 常见错误

### 错误 1：并行化杀死缓存

**现象：** 启用提示词缓存后账单没有下降，反而增加了。

**原因：** 10 个并行工具调用同时到达，全部缓存未命中，全部支付写入溢价。

**修复：** 先顺序发出第一个请求让缓存填充，再并行发出其余。

### 错误 2：动态内容在前缀中

**现象：** 提示词缓存命中率只有 7%。

**原因：** 系统提示词包含日期、请求 ID 等动态内容——每个请求都不同。

**修复：** 将静态部分移到可缓存前缀，动态部分追加到缓存边界之后。

## 7. 面试考点

### Q1：L2 缓存和 L1 语义缓存的区别是什么？（难度：⭐⭐）

**参考答案：**
L2（提供商级）缓存：提供商存储前缀的 KV 缓存，下一个匹配前缀的请求直接复用——不需要重新 prefill。Anthropic 和 OpenAI 原生支持。L1（应用级）语义缓存：在调用 LLM 之前，将提示词嵌入并与缓存中的嵌入比较——余弦相似度 > 0.95 时直接返回缓存响应。L2 只对完全相同的前缀有效；L1 可以对语义相似但字面不同的提示词命中。两者可以叠加使用。

### Q2：如何将提示词缓存命中率从 7% 提升到 74%？（难度：⭐⭐⭐）

**参考答案：**
ProjectDiscovery 的案例：将系统提示词中的动态内容（日期、请求 ID、随机化示例）移到可缓存前缀之后。静态部分（系统角色、规则、示例）放在缓存边界之前，动态部分追加在后面。这样所有请求共享相同的前缀——缓存命中率从 7%（每个请求都不同）提升到 74%（前缀相同的部分被缓存）。

## 📚 小结

提示词缓存在两个层面发生：L2（提供商级 KV 复用）和 L1（应用级语义缓存）。Anthropic 的缓存读取比新鲜输入便宜 10 倍；OpenAI 自动缓存对 ≥1024 词元的提示词生效。两个陷阱：并行化（账单膨胀 5-10 倍）和动态内容（命中率崩溃）。ProjectDiscovery 的案例证明：将动态内容移出前缀，命中率从 7% 提升到 74%。批处理 + 缓存叠加可以将夜间工作负载的成本降到基线的约 10%。
