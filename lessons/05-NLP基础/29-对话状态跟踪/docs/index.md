# 对话状态跟踪

> "我想要北区便宜点的餐馆…算了改成中等价位…再加意大利菜。"三轮对话、三次状态更新。DST 保持槽位-值字典的同步——让预订操作在执行时拿到正确的参数。

## 1. 问题

在任务导向对话系统中，用户的目标被编码为一组**槽位-值对**：`{菜系: 意大利, 区域: 北区, 价格: 中等}`。每一轮用户对话都可能增加、修改或删除一个槽位。系统必须阅读整个对话历史，正确输出当前状态。

搞错一个槽位 → 系统订了错误的餐厅、安排了错误的航班、或扣了错误的卡款。DST 是**用户说了什么**和**后端执行了什么**之间的铰链。

**为什么 2026 年 LLM 时代仍然重要：**
- 合规敏感域（银行、医疗、航空订票）要求确定性槽位值——不是自由形式生成
- 工具调用智能体在调用 API 之前仍需要槽位解析
- 多轮修正比看起来更难："算了，改成周四吧。"

---

## 2. DST 的核心操作

```
Turn 1: "我想找北区便宜的餐馆"
  State: {area: north, price: cheap}

Turn 2: "改成中等价位的"
  State: {area: north, price: moderate}  ← price 被覆盖

Turn 3: "再加意大利菜"
  State: {area: north, price: moderate, cuisine: italian}  ← 新增
```

三种状态更新操作：
- **ADD**——新增一个之前未设置的槽位
- **UPDATE**——覆盖已有槽位的值
- **DELETE**——"不需要了"——清除槽位

---

## 3. 中文 DST 特别挑战

- **中文的口语化价格表达。** "便宜点的" = cheap、"差不多就行" = moderate、"贵一点的" = expensive——这些表达在英文中是明确的 price range 关键词（cheap/moderate/expensive），中文需要更宽泛的语义映射
- **中文的"可以/行/好"不总是确认。** "中等价位行吗？"——"行"表示确认（update price→moderate）。"意大利菜可以吗？"——"都可以"表示不指定（不 update cuisine）。区分"确认"和"不关心"是中文对话中 DST 最大的难点
- **LLM-based DST：** 用提示词指示 LLM 从对话历史中提取 slot-value JSON——给出当前状态 + 本轮对话 → 返回更新后的状态。在开放域表现好，但在合规域中，始终用规则确认破坏性槽位更新（付款金额、日期、身份证号）
