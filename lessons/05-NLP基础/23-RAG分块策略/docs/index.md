# RAG 分块策略

> 分块配置对检索质量的影响与嵌入模型选择相当。分块做错了，再多的重排也救不回来。

**类型：** 实现课 | **语言：** Python | **前置知识：** 阶段 05 · 14、05 · 22 | **预计时间：** ~60 分钟 | **所处阶段：** Tier 1

## 🎯 学习目标

## 1. 问题

你把一份 50 页的合同放进 RAG 系统。用户问："终止条款是什么？"检索器返回了封面页。为什么？因为模型用 512-token 的块训练，终止条款在第 20 页——跨过了页码分隔，周边没有与查询匹配的局部关键词。

修复不是"买个更好的嵌入模型"。修复是分块——多长？重叠多少？在哪切？要不要带上下文？

**2026 年 2 月的基准给出了意外结论：**
- 递归 512-token 分块打败了语义分块（69% → 54% 准确率）
- SPLADE + Mistral-8B 在 Natural Questions 上，重叠提供了零可测量的收益
- 上下文悬崖：响应质量在约 2500 token 的上下文处急剧下降

---

## 2. 五种分块策略

| 策略 | 做法 | 适用 |
|---|---|---|
| **固定大小** | 每 N token，重叠 M token | 基线。1000 token 块 + 200 重叠是最常见的起点 |
| **递归** | 按 `\n\n` → `\n` → `。` → ` ` 优先级逐级分割 | 通用文档。2026 基准的赢家 |
| **语义** | 用嵌入模型的相似度变化来检测"主题切换点" | 理论上最优。实践中意外地弱 |
| **文档结构** | 按 Markdown 标题、HTML 标签、PDF 章节切分 | 结构化文档。元数据丰富但有结构断裂点 |
| **句子窗口** | 每句一个块 + 嵌入时带周围句子的上下文 | 精准匹配 + 不丢上下文。LLM 重排前的候选 |

---

## 3. 中文分块特别建议

- **按句号分块优于按字符数。** 中文句号（`。`）是自然的语义边界——比按 512 字硬切更能保留完整的语义单元。使用"句子数为单位"而非"字符数为单位"做中文分块
- **中文的递归优先级：** `\n\n` → `\n` → `。` → `；` → `，`。最后一步才考虑按"字符数"硬切
- **中文重叠用"句数"而非"token 数"：** 重叠 2-3 个句子比重叠 200 个 token 更稳定——因为中文字符/token 比例在不同文本中差异很大（文言文 vs 网络口语）
- **中文 Markdown 文档的标题分块：** 中文技术文档（如本项目的 Lesson 文档）天然有 `##` 标题结构——按 `##` 分块 + 保留上级标题作为元数据 = 最有效的中文文档分块方案

---

## 4. 分块配置检查清单

- [ ] 块大小在嵌入模型的上下文窗口之内（标称的 70%）
- [ ] 重叠确保一个句子不会被切成两半（至少重叠 1-2 个完整句）
- [ ] 每个块携带了它的来源元数据（文档名、页码、章节标题）
- [ ] 在目标领域的 20 个真实查询上测了 Recall@5

---

## 📚 小结

分块策略是 RAG 流水线中最被低估的环节。递归按标点优先级切割（而非语义切割）是 2026 基准的意外赢家。中文按句号+句子数分块，重叠以完整句子为单位。
