# 关系抽取与知识图谱构建

> NER 找到了实体。实体链接锚定了它们。关系抽取找到了它们之间的边。知识图谱是节点、边及其出处的总和。

**类型：** 实现课 | **语言：** Python | **前置知识：** 阶段 05 · 06（NER）、05 · 25（实体链接） | **预计时间：** ~60 分钟 | **所处阶段：** Tier 1

---

## 🎯 学习目标

- [ ] 理解关系抽取如何将自由文本转换为结构化三元组 (主语, 关系, 宾语)
- [ ] 区分预定义关系抽取（分类）和开放关系抽取（从文本中提取出关系短语本身）
- [ ] 理解知识图谱在 RAG 中的应用——结构化的事实核查比向量相似度更可靠

## 1. 问题

分析师读到："Tim Cook 于 2011 年成为苹果公司 CEO。"四个事实：

- (Tim Cook, 职位, CEO)
- (Tim Cook, 雇主, 苹果)
- (Tim Cook, 入职时间, 2011)
- (苹果, 类型, 公司)

关系抽取（Relation Extraction, RE）将自由文本转化为结构化三元组 `(主语, 关系, 宾语)`。在语料库上聚合 → 知识图谱。聚合 + 查询 → RAG 的推理基础、分析或合规审计。

---

## 2. 两种范式

| | 预定义 RE | 开放 RE |
|---|---|---|
| **关系类型** | 预先定义好的集合（雇主、创始人、总部…） | 从文本中提取出关系短语本身 |
| **建模方式** | 多分类（或 NLI：蕴含=关系成立） | Seq2Seq 生成 / 三元组抽取 |
| **适用场景** | 领域明确、关系类型已知 | 开放域、未知关系类型 |
| **中文模型** | `bert-base-chinese` + 关系分类头 | HanLP OpenRE、LLM few-shot |

---

## 3. 知识图谱 + RAG——2026 应用模式

```
文档 → NER → 实体链接 → 关系抽取 → 知识图谱
                                            ↓
查询 → 在 KG 中查找同事实的实体和关系 → 结构化证据 → LLM 回答
```

KG 提供的事实核查比向量相似度更可靠——一个三元组要么在 KG 中（事实），要么不在（需要验证）。不存在"50% 相似"的模糊匹配。中文金融/法律知识图谱（如 CN-DBpedia、OwnThink）是中文 RAG 系统中结构化事实核查的直接来源。
