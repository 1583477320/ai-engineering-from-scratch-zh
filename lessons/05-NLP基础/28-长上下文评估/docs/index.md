# 长上下文评估——NIAH、RULER、LongBench、MRCR

> Gemini 3 Pro 标称 10M token 上下文。在 1M token 处，8 针 MRCR 降到 26.3%。标称 ≠ 可用。长上下文评估告诉你实际可用的容量。

## 🎯 学习目标

## 1. 问题

你有 200 页合同。模型声称支持 1M token 上下文。你贴进去问"终止条款是什么？"——模型从封面页回答了。因为终止条款在 120k token 深处——超出了模型实际能注意到的范围。

**2026 年的上下文容量鸿沟：** 规格表说 1M 或 10M。现实说 60-70% 可用——且"可用"依赖于任务：
- 检索（单针 NIAH）：前沿模型在标称最大值处近乎完美
- 多跳/聚合：在大多数模型上 ~128k 后急剧退化
- 分散事实的推理：第一个失败的任务

---

## 2. 四个基准

| 基准 | 测什么 | 2026 前沿结果 |
|---|---|---|
| **NIAH**（大海捞针） | 128k 文档某处放一个事实——能找到吗？ | 近乎完美直到标称最大值 |
| **RULER** | 多针+多跳。NIAH 升级——不只找针，还要综合多针信息 | 远低于标称。多跳在 128k+ 急剧退化 |
| **LongBench** | 多任务长文理解——QA、摘要、少样本 | 各模型差异极大 |
| **MRCR**（多针复述） | 8 根针分散——能全部复述吗？ | Gemini 3 Pro 10M→1M 处仅 26.3% |

**关键发现：分散事实的推理是第一个失败的任务。**

## 3. 构建领域针测试

```python
def needle_test(model, doc, needle, position_ratio):
    insert_pos = int(len(doc) * position_ratio)
    modified = doc[:insert_pos] + f"\n{needle}\n" + doc[insert_pos:]
    answer = model.query(modified, "针的内容是什么？")
    return needle.lower() in answer.lower()

# 在 0%, 25%, 50%, 75%, 100% 五个位置各测
for ratio in [0.0, 0.25, 0.5, 0.75, 1.0]:
    print(f"位置 {ratio:.0%}: {needle_test(model, doc, needle, ratio)}")
```

**真实验 = 你的领域文档 + 你的任务。** 中文注意：中文 token vs 英文 token 约 1:1.5——标称 128k ≈ 85k 中文字。针测试结果的跨语言迁移有限——中文需独立测试。

---

## 4. 陷阱

- **NIAH 单针通过 ≠ 长上下文"能用"。** NIAH 是最低门槛——检索能力。推理/聚合/比较是真正区分模型的任务
- **MRCR 揭示的问题比 NIAH 更接近真实使用。** 用户在一个文档中问不止一个问题——分散的多针检索是 2026 生产场景的常态

---

## 🔑 关键术语 | 📚 小结

四个基准测不同维度：NIAH（最低门槛——检索）、RULER（多跳推理）、LongBench（多任务理解）、MRCR（多针复述——最接近生产场景）。"标称"≠"可用"——实际可用 = 标称 × 60-70%，且依赖于任务。始终在领域文档上构建针测试。

## ✏️ 练习

1. 【理解】在  模型上跑 NIAH——在 128k 上下文 0/25/50/75/100% 位置各测一根针。
2. 【实现】构建多针 MRCR 测试——8 根针分散在文档中——衡量多针复述准确率 vs 单针。
3. 【思考】你的长文档 QA 系统在 NIAH 上完美但在真实查询上很差——NIAH 漏掉了什么？
