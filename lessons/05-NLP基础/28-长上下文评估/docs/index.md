# 长上下文评估——NIAH、RULER、LongBench、MRCR

> Gemini 3 Pro 标称 10M token 上下文。在 1M token 处，8 针 MRCR 降到 26.3%。标称 ≠ 可用。长上下文评估告诉你实际可用的容量。

## 1. 问题

你有一份 200 页的合同。模型声称支持 1M token 上下文。你把合同贴进去问："终止条款是什么？"模型回答了——但从封面页回答了。因为终止条款在 120k token 深的地方——超出了模型实际能注意到的范围。

**2026 年的上下文容量鸿沟：** 规格表说 1M 或 10M。现实说 60-70% 可用——而且"可用"依赖于任务类型。

---

## 2. 四个基准——各量不同维度

| 基准 | 测什么 | 典型结果（2026 前沿模型） |
|---|---|---|
| **NIAH**（大海捞针） | 128k 文档中某处放一个事实。模型能找到吗？ | 前沿模型在标称最大值处近乎完美 |
| **RULER** | 多针 + 多跳推理。NIAH 的升级版——不只找到针，还要综合多根针的信息 | 远低于标称最大值。多跳在 128k+ 急剧退化 |
| **LongBench** | 多任务长文本理解——QA、摘要、少量样本学习 | 各模型差异极大——不存在"最好"的模型 |
| **MRCR**（多针复述） | 8 根针分散在文档中——模型能全部复述吗？ | Gemini 3 Pro 10M → 1M 处仅 26.3% |

**关键发现：分散事实的推理是第一个失败的任务。**

## 3. 构建领域针测试

```python
def needle_test(model, doc, needle, position_ratio):
    """在文档的 position_ratio 处插入 '针' → 问模型针的内容。"""
    insert_pos = int(len(doc) * position_ratio)
    modified = doc[:insert_pos] + f"\n{needle}\n" + doc[insert_pos:]
    answer = model.query(modified, "针的内容是什么？")
    return needle.lower() in answer.lower()
```

- 在 0%、25%、50%、75%、100% 五个位置各测试
- 真实验：你的领域文档 + 你的任务。通用基准的结论不一定迁移到你的分布上
- 中文注意：中文 token 与英文 token 的比例约 1:1.5——一个中文字 ≈ 1.5 个英文 token。标称 128k 的中文语境大约等于 85k 个中文字
